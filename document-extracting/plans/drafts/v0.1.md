# MOI vs LandingAI 文档信息提取产品对比测评计划

> 状态：Draft v0.1，待 MOI 接口确认、LandingAI 账号/条款确认和 Smoke Test 后冻结
>
> 对比对象：MOI 文档信息提取、LandingAI Agentic Document Extraction
>
> 评测任务：给定相同文档和预定义业务 Schema，比较两个产品的字段提取效果与使用成本
>
> 正式规模：SROIE、VRDU Registration、Kleister-NDA 各 100 份，共 300 份
>
> 计划日期：2026-07-23

## 1. 测评目标

本轮不是单独验证 LandingAI，而是在相同输入、相同业务字段和相同评分规则下，对比 MOI 与 LandingAI，回答四类问题：

1. **提取质量**：哪个产品的字段准确率、完整率更高，幻觉更少？
2. **场景能力**：两个产品在低质量收据、多模板表单、多页法律文档上的优劣分别是什么？
3. **工程可用性**：Schema 适配、批量运行、异常处理、结果溯源和人工复核能力有何差异？
4. **效率与成本**：完成同一批 300 份文档，各自需要多少时间、调用成本和人工操作？

最终产物应支持产品选型和 MOI 能力改进，而不是只给出一个无解释的综合分。

## 2. 对比对象

| 对比项 | MOI | LandingAI |
|---|---|---|
| 定位 | 我方文档信息提取产品 | 外部商业对比产品 |
| 输入 | 原始 PDF/JPG/PNG | 原始 PDF/JPG/PNG |
| 提取要求 | 预定义业务 Schema | 同一份预定义业务 Schema |
| 输出 | 结构化字段；同时保存产品提供的定位、置信度等信息 | Extract 结构化字段；同时保存 Grounding、confidence 等信息 |
| 调用方式 | 以正式可用的 API/产品流程为准，Smoke 前冻结版本与配置 | ADE Parse + Extract，优先使用异步 Standard Jobs |
| 调参原则 | 仅在 Smoke/Pilot 阶段统一调整 | 仅在 Smoke/Pilot 阶段统一调整 |
| 正式运行 | 冻结配置，一份文档一次有效业务响应 | 冻结配置，一份文档一次有效业务响应 |

### 2.1 公平性原则

- 两个产品使用完全相同的 300 份源文件和 Golden；
- 每个数据集共用同一份字段定义、字段类型、字段描述和缺失值语义；
- 若产品 Schema 语法不同，只允许做无损语法映射，不得增删业务信息；
- 不把某一产品的解析结果作为另一产品的输入；
- Adapter 只转换格式，不修复或补全产品输出；
- Schema、Prompt 和配置在正式运行前冻结，不按正式 Case 逐题调优；
- 基础设施失败可以按统一规则重试，字段提取错误不得重试选优；
- 两个产品均报告首次有效结果，不从多次调用中挑最好结果；
- 记录实际产品版本、模型版本、运行时间和配置，避免版本漂移导致不可比。

## 3. 测试数据集

### 3.1 总体情况

| 数据集 | Case | 预计页数 | 固定业务字段 | 文档特征 | 主要考察能力 |
|---|---:|---:|---:|---|---|
| SROIE | 100 | 100 | 4 | 单页扫描收据 | OCR、字段定位、相似金额区分 |
| VRDU Registration | 100 | 202 | 6 | 同一业务、三种模板、多页表单 | 跨模板泛化、位置变化、缺失字段 |
| Kleister-NDA | 100 | 602 | 4（含数组） | 多页法律合同 | 长文档检索、跨页语义、多值与归一化 |
| **合计** | **300** | **约 904** | — | 三类公开文档 | 从视觉识别到长文档语义提取 |

选择这三个数据集的原因是：它们都能为同类文档定义统一 Schema，同时覆盖不同难度和不同业务形态。

不纳入首版的数据集：

- FUNSD：199 份文件属于不同表单，业务字段不统一，不适合一个 Schema 批量运行；
- CORD：与 SROIE 同属收据场景，当前数据质量和嵌套行项目适配成本不符合首版目标；
- VRDU Ad-buy Form：包含更复杂的组合/行项目问题，留作后续扩展；
- Kleister-NDA test-A：Golden 不公开，无法进行本地一致评分。

### 3.2 SROIE：扫描收据

本地数据：`datasets/SROIE2019/`

正式集从带公开 Golden 的 train 中按固定随机种子选择 100 份，并冻结文件名和 SHA-256。

字段 Schema：

```json
{
  "company": "string",
  "date": "string",
  "address": "string",
  "total": "string"
}
```

重点测试：

- 低清晰度、倾斜、模糊、小字号下的字段识别；
- 多行地址是否完整；
- 日期字符是否识别正确；
- 是否把 subtotal、tax、cash、change 误识别为 total；
- 商户名称和地址中的 OCR 错误。

### 3.3 VRDU Registration：多模板注册表单

本地数据：`datasets/VRDU/registration-form/`

正式使用官方划分文件：

```text
FARA-lv2-mixed_template-train_10-test_300-valid_100-SD_0.json
```

取其中 `valid` 的 100 份，覆盖 Amendment、Short Form 和 Dissemination Report 三类模板。

字段 Schema：

```json
{
  "file_date": "string(date)",
  "foreign_principle_name": "string",
  "registrant_name": "string",
  "registration_num": "string",
  "signer_name": "string",
  "signer_title": "string"
}
```

`foreign_principle_name` 保留官方数据集拼写，避免评分键不一致。

重点测试：

- 同一业务字段在不同模板中的标签和位置变化；
- 字段跨页或远离标签；
- 同一值在文档中多次出现时能否选对；
- 缺失字段是否正确返回空值，而不是幻觉；
- registrant、foreign principal、signer 等相似实体是否混淆。

### 3.4 Kleister-NDA：多页法律合同

本地数据：`datasets/Kleister-NDA/`

正式 100 份由以下部分构成：

- `dev-0` 全部 83 份；
- 从 `train` 按固定随机种子抽取 17 份；
- 不使用 Golden 隐藏的 `test-A`。

字段 Schema：

```json
{
  "effective_date": "string(date)",
  "jurisdiction": "string",
  "party": ["string"],
  "term": "string(normalized duration)"
}
```

重点测试：

- 关键信息位于长文档中间或签署页；
- 生效日期与签署日期混淆；
- governing law、venue、地址中的州名混淆；
- 合同主体与签署人混淆；
- 多方合同的数组提取；
- 从法律表述中推断并归一化合同期限；
- decoy key 和缺失字段导致的误填。

## 4. 数据准备

### 4.1 冻结 Manifest

每个 Case 至少记录：

```yaml
case_id: vrdu_0001
dataset: vrdu_registration
split: valid
source_path: datasets/VRDU/registration-form/main/pdfs/example.pdf
source_sha256: sha256:...
page_count: 2
schema_version: vrdu_registration_v1
gold_source: dataset.jsonl
```

正式运行前冻结三个 100 Case Manifest、源文件哈希、Schema 和 Golden Adapter 版本。

### 4.2 Golden Adapter

```text
SROIE entities JSON
  -> {company, date, address, total}

VRDU annotations
  -> {file_date, foreign_principle_name, registrant_name,
      registration_num, signer_name, signer_title}

Kleister expected.tsv
  -> {effective_date, jurisdiction, party[], term}
```

Adapter 只做格式解析、空值统一、数组转换和官方规范化规则实现，不得参考 MOI 或 LandingAI 的输出修改 Golden。

### 4.3 产品输出 Adapter

两个产品分别保留原始响应，再映射为统一评分结构：

```json
{
  "case_id": "vrdu_0001",
  "system": "moi | landingai",
  "fields": {},
  "schema_valid": true,
  "grounding": {},
  "confidence": {},
  "latency_ms": 0,
  "usage": {},
  "error": null
}
```

无法解析、超时、服务错误和 Schema 不合法必须显式记录，不能静默丢弃。

## 5. 评测方面

### 5.1 字段提取效果：核心指标

每个数据集、每个产品分别报告：

- 字段级 Precision、Recall、F1；
- Normalized Exact Match；
- 文档全字段正确率；
- 缺失字段正确率；
- 幻觉率；
- Schema 合法率；
- API/任务成功率。

三个数据集差异较大，不直接混合全部字段计算 micro 总分。总体指标采用数据集等权宏平均：

```text
Overall Macro F1 =
  (SROIE F1 + VRDU F1 + Kleister-NDA F1) / 3
```

综合分只用于概览，产品判断必须同时查看三个数据集和逐字段结果。

### 5.2 归一化与匹配

通用规则：

- Unicode 规范化、去首尾空白、合并连续空格；
- 忽略大小写，但不删除改变语义的数字、单位和否定词；
- 同时保留 Raw Exact Match 和 Normalized Match。

专项规则：

| 字段 | 评分规则 |
|---|---|
| SROIE `address` | 换行转空格并合并空白，保留字母和数字顺序 |
| SROIE `total` | 去货币符号、千位分隔符后按数值比较 |
| 日期 | 解析后统一为 `YYYY-MM-DD` 比较 |
| VRDU 姓名 | 使用官方 NameMatch 等价规则或经验证的本地实现 |
| `registration_num` | 按字符串比较，保留前导零 |
| Kleister `party` | 集合级 P/R/F1，不依赖返回顺序 |
| Kleister `term` | 统一数字、单复数和时间单位 |
| 缺失字段 | Golden 缺失时，仅 `null`、缺键或空数组按约定判对 |

### 5.3 场景能力与错误分析

除主分数外，按以下维度切片：

- SROIE：清晰度、倾斜、字段长度、金额干扰；
- VRDU：三种模板、单页/多页、字段缺失与字段重复；
- Kleister：页数分桶、字段所在页、单值/多值、显式/隐式表达；
- 逐字段准确率；
- OCR 错误、定位错误、实体混淆、语义推断错误、归一化错误、幻觉；
- 两个产品一对一 Case 差异：MOI 独对、LandingAI 独对、同时正确、同时错误。

对结果差异使用配对统计。字段正确性为二元结果时使用 McNemar 检验，并给出 bootstrap 95% 置信区间；不只比较两个点估计。

### 5.4 工程与产品能力

| 方面 | 观察内容 |
|---|---|
| Schema 能力 | 类型、数组、字段描述、日期/格式、缺失值表达 |
| 批处理 | 并发、队列、限流、超时、恢复和断点续跑 |
| 可追溯性 | 页码、Bounding Box/Grounding、置信度、原文证据 |
| 易用性 | Schema 配置工作量、API/界面操作、错误信息质量 |
| 稳定性 | 首次成功率、重试率、输出结构一致性 |
| 可观测性 | 请求 ID、模型版本、usage、错误码和日志 |
| 数据与合规 | 数据保留、区域、隐私选项、合同限制 |

其中“易用性”保留客观记录，例如配置步骤数、人工分钟数和失败次数；主观评价需单独标记，不混入字段准确率。

### 5.5 性能与成本

每个产品记录：

- 端到端 P50、P95 时延；
- 总运行时间和有效吞吐；
- 每文档、每页平均成本；
- 正式集总 API/服务成本；
- Schema 配置、运行监控和异常处理的人工时间；
- 若 MOI 为自部署，另记计算资源规格和资源成本，不与纯 API 费用混为一项。

## 6. 运行流程

### 6.1 Smoke：9 份

每个数据集选择 3 份，共 9 份，同时运行 MOI 和 LandingAI：

- 1 份常规样本；
- 1 份版式或质量较难样本；
- 1 份包含缺失、多值或易混淆字段的样本。

验证文件输入、Schema、响应结构、缺失值、数组、日期、证据定位、版本信息、限流和 usage。Smoke 后可以做一次统一 Schema 修订。

### 6.2 Pilot：30 份

每个数据集 10 份，共 30 份，用于：

- 验证两个产品 Adapter 和统一 Scorer；
- 校验相同输入、相同 Schema 语义和运行参数；
- 外推时延、成本和并发设置；
- 检查结果落盘、错误分类和断点续跑；
- 形成第一版差异 Case 人工复核规范。

如果 Pilot 用于调参，其 Case 不进入正式统计，并从同分层备用样本替换。

### 6.3 正式运行：300 份 × 2 个产品

1. 冻结 Manifest、Golden、Schema、Adapter、Scorer 和两个产品配置；
2. 对 MOI 运行全部 300 份并保存原始响应；
3. 对 LandingAI 运行同一批 300 份并保存 Parse/Extract 原始响应；
4. 只对网络、限流和服务端错误按统一规则重试；
5. 运行统一 Adapter 和 Scorer；
6. 自动生成总体、数据集、字段、场景和 Case 配对结果；
7. 人工复核全部产品分歧 Case，或数量过大时按错误类型分层抽样；
8. 形成结论、优势场景、短板和 MOI 改进建议。

为了减少时间段和外部服务负载差异，两个产品按数据集交替、分批运行。若运行期间产品版本变化，停止当前 Run，不能将不同版本结果混在一起。

### 6.4 重试口径

允许重试：

- HTTP 408、429、502、503、504；
- 明确的服务端失败；
- 上传中断或结果下载失败。

不允许重试：

- 字段值错误或返回 `null`；
- OCR、定位、语义或格式错误；
- 合法业务响应中的字段遗漏；
- 为取得更好结果而重复调用。

报告首次成功率和重试后的最终成功率；主质量分采用第一次有效业务响应。

### 7.2 四个结论方面

1. **准确性对比**：总体、逐数据集、逐字段、缺失值、多值字段和文档全对率；
2. **能力边界对比**：低质量 OCR、跨模板、多页检索、语义推断和幻觉；
3. **工程体验对比**：Schema 表达、批量运行、稳定性、可追溯性和错误处理；
4. **效率成本对比**：速度、API/资源成本、人工投入和规模化运行成本。

结论采用“数据 + 代表 Case + 原因分析”的方式。例如不能只写“MOI 在 Kleister 更好”，而应写出差值、置信区间、主要获胜字段及典型错误。

### 7.3 不做单一加权总排名

首版不将准确率、成本、时延和易用性混成一个人为加权总分。报告给出：

- 质量主结果；
- 各业务场景胜负；
- 性能和成本；
- 工程能力；
- 适用场景与限制。

如后续业务明确给出权重，再生成面向具体选型场景的加权结论。

## 8. LandingAI 账号与预算

截至 2026-07-23 的官方公开计费口径：

- 1 credit = 0.01 美元；
- Explore 新账号含 1,000 免费 credits，90 天过期；
- Explore 可按量购买，无需购买 Team 套餐；
- 异步 Standard Jobs 约为 Priority 的一半。

按当前 300 份、约 904 页的本地规模估算，LandingAI 正式运行约需 1,621 credits。加上 Smoke、Pilot、Schema 调试和基础设施重试，建议准备：

```text
2,000～2,500 credits
= 20～25 美元
```

若账号仍有完整 1,000 免费 credits，预计充值约 10～15 美元。最终以 9 Case Smoke 的实际 usage 外推为准。

LandingAI 服务条款包含与 benchmarking/competitive analysis 相关的限制。正式执行产品对比、尤其是公开发布结果前，需由公司确认合同权限或取得书面许可；未确认前只进行内部小规模功能验证。

MOI 的调用成本、部署资源和内部计费方式在 Smoke 时补齐，最终报告必须使用同一成本边界说明。

## 10. 风险与限制

- 三个数据集均为公开集，可能存在训练数据泄漏；结果只能表述为公开集对比，不能直接外推到私有业务文档；
- SROIE、VRDU 的 Golden 可能存在 OCR、拼写和标注噪声；
- Kleister 的值有特殊归一化，100 份为本项目构造的公开有标签子集，不是官方 test-A；
- 两个产品的底层解析链路不同，这是产品端到端能力的一部分，但报告需说明差异；
- 商业产品的价格、限流和模型别名可能变化，正式运行前需重新核实；
- 争议 Golden 单独标记，分别保留原始 Golden 分数和审计后分数，不静默修订；
- 若产品输出的证据定位能力不同，只做独立能力比较，不因缺少可选证据字段直接降低核心字段 F1。
