# MOI RAG 端到端 Benchmark 计划

## 1. 评测目标

使用同一批原始非结构化文件，分别在 MOI 和竞品中独立完成建库，由各产品使用原生的解析、清洗、分段、Embedding、索引、检索、重排和生成流程，比较最终用户可感知的 RAG 问答效果。

本轮属于**产品原生端到端评测（Native End-to-End Track）**。结果代表产品完整 RAG 管线的综合能力，不应直接解释为某个单独解析器、Embedding 模型或检索算法的优劣。

## 2. 评测对象

第一阶段：

- MOI
- Dify
- FastGPT

第二阶段增强项：

- RAGFlow

RAGAS、DeepEval属于评测工具，不作为竞品。

## 3. 公平性控制

1. 所有产品导入完全相同的原始文件，不复用此前数据库中的现成 chunk 或向量作为正式评测输入。
2. 在运行各产品之前，直接依据原始文件构建并冻结测试集；不能使用 MOI 或任一竞品的回答生成正式 Golden。
3. 在产品允许配置时，统一生成模型、模型版本、温度、最大输出长度和系统指令；无法统一的部分明确记录为产品原生配置。
4. 每个产品允许使用开发集进行一次合理调参，记录分段、Embedding、Top-K、混合检索、rerank和Prompt配置；配置冻结后才能运行隐藏测试集。
5. 所有产品在同一时间窗口和可比资源条件下运行。每条问题至少重复3次，质量指标报告均值，时延报告P50/P95。
6. 记录产品版本、部署方式、硬件资源、模型/API版本、建库时间、调用成本、重试和失败样本。
7. 内部文件必须经过数据安全确认。如果不允许上传第三方SaaS，则使用私有化部署，或另建可外发的脱敏语料组。

## 4. 测试集构建计划

### 4.1 测试集规模

首版建议自动构建150～300条问题，不安排逐题人工审核。生成后通过去重、证据包含、答案可支持性、格式完整性和泄漏检查等程序化规则筛选并冻结。

### 4.2 构建步骤

1. 盘点并冻结原始文件清单，计算文件哈希。
2. 记录文件类型、页数、语言、业务领域、版本日期，以及是否包含表格、图片、公式等内容。
3. 从原始文件抽取可验证事实和证据片段，由LLM生成候选问题、参考答案、相关文件和证据位置。
4. 去除重复、歧义、依赖外部常识、无法从文件独立回答的问题。
5. 使用程序化规则和独立模型检查问题、参考答案与证据的一致性，过滤未通过校验的样本。
6. 按文档维度切分开发集和隐藏测试集，建议比例为20%/80%。同一文档生成的问题不能同时出现在两部分中。
7. 冻结测试集版本；正式评测期间不得根据产品输出修改Golden。

### 4.3 问题类型

| 类型 | 建议占比 | 主要考察能力 |
|------|:-------:|-------------|
| 单文档单证据事实题 | 30% | 基础解析、召回和事实回答 |
| 单文档跨段综合题 | 20% | 长文档定位与多段综合 |
| 跨文档多跳题 | 15% | 多文档召回与信息组合 |
| 表格/图片相关题 | 10% | 复杂非结构化内容处理；仅在语料包含时启用 |
| 精确关键词题 | 10% | 编号、型号、日期、专有名词和错误码检索 |
| 口语化、缩写和改写题 | 10% | Query Rewrite和语义鲁棒性 |
| 知识库无答案题 | 5% | 拒答、幻觉控制和错误引用防护 |

### 4.4 测试样本格式

每条样本至少包含：

```json
{
  "question_id": "rag-0001",
  "question": "用户问题",
  "reference_answer": "由证据生成并通过自动校验的参考答案",
  "evidence_text": ["支持答案的原文证据"],
  "source_file_ids": ["document-001"],
  "evidence_locations": ["第3页/第二节"],
  "question_type": "single_document_multi_evidence",
  "difficulty": "medium",
  "answerable": true,
  "generator": "model-and-prompt-version",
  "validation_status": "passed"
}
```

如果能够稳定取得原文分段ID，可以额外保存`relevant_chunk_ids`，用于检索诊断。

## 5. 评测指标

### 5.1 端到端回答质量：主指标

| 指标 | 定义与评分方法 | 适用范围 |
|------|---------------|---------|
| 答案正确性 Answer Correctness | 回答与人工参考答案在关键事实上的一致程度；LLM Judge评分后抽样人工复核 | 可回答问题 |
| 忠实度 Faithfulness | 回答中的事实是否全部能由该产品实际引用或召回的上下文支持 | 可回答问题 |
| 答案完整性 Answer Completeness | 参考答案中的关键要点被覆盖的比例；多要点题逐项计分 | 可回答问题 |
| 引用正确率 Citation Precision | 被引用证据中真正支持回答陈述的比例 | 返回引用的产品 |
| 引用覆盖率 Citation Recall | 需要证据支持的关键陈述中，具有正确引用的比例 | 返回引用的产品 |
| 无答案正确率 | 无答案问题中正确拒答且未编造事实的比例 | 无答案问题 |
| 任务成功率 | 同时达到正确性、忠实度和最低引用要求的样本比例 | 全部问题 |

推荐首版主结果表至少包含：Answer Correctness、Faithfulness、Answer Completeness、无答案正确率、任务成功率、P95端到端时延、查询成功率和单次查询成本。

### 5.2 检索与证据质量：诊断指标

仅在产品能够导出召回内容时统计。不能导出时标记为N/A，不能通过最终答案反推召回结果。

| 指标 | 定义 |
|------|------|
| Document Recall@K | Gold相关文件是否出现在前K个召回结果中 |
| Evidence/Chunk Recall@K | Gold证据是否被前K个结果覆盖 |
| MRR | 第一个相关结果排名的倒数均值 |
| nDCG@K | 多个相关证据的整体排序质量 |
| Context Precision | 提供给生成模型的上下文中相关内容所占比例 |

各产品的分段边界不同，Evidence/Chunk Recall不能只比较chunk ID。应优先依据来源文件、页码/章节，以及Gold证据文本的文本或语义覆盖关系判断命中。

### 5.3 性能、成本与工程指标

| 指标 | 统计方法 |
|------|---------|
| 建库成功率 | 成功处理文件数/总文件数，并按文件类型统计失败率 |
| 建库时间 | 从提交文件到全部文件可查询的总耗时 |
| 索引规模 | chunk数、索引占用空间、Embedding/API用量 |
| 首字时延TTFT | 发起问题到返回首个token的时间 |
| 端到端时延 | 发起问题到完整答案返回的时间，报告P50/P95 |
| 查询成功率 | 正常返回有效答案的请求数/总请求数 |
| 单次查询成本 | Embedding、rerank、LLM和平台API成本合计 |
| 人工配置成本 | 从创建知识库到可运行评测所需的人时和人工步骤数 |

## 6. 评分与人工复核

1. 以自动指标和固定 LLM Judge 为主，并冻结 Judge 模型、Prompt、温度和版本。
2. 只对 Judge 不一致、系统分差很小或明显异常的结果做少量人工抽查，不设固定双人盲审比例。
3. 人工抽查时隐藏 MOI 和竞品名称。
4. 报告主结果及必要的分组结果，不开展大规模人工评分。

## 7. 执行流程

1. 冻结语料清单并生成测试集Draft。
2. 完成自动校验，切分开发集和隐藏测试集。
3. 将相同原始文件分别导入MOI、Dify和FastGPT，记录建库过程和失败文件。
4. 仅使用开发集完成各产品合理调参，然后冻结全部配置。
5. 使用统一调用适配器运行隐藏测试集，保存问题、回答、引用、召回内容、耗时、错误和成本。
6. 运行自动评分，仅对争议或异常结果进行少量匿名人工抽查。
7. 输出主结果表、少量代表性失败案例、配置清单和可复现运行记录。

## 8. 交付物

- 冻结的原始语料清单和文件哈希
- 自动生成并通过程序化校验的开发集与隐藏测试集
- MOI、Dify、FastGPT的版本与配置清单
- 统一产品调用Adapter和原始运行结果
- 自动评分结果与少量争议样本抽查记录
- Benchmark 主结果、必要分组结果和代表性失败案例
