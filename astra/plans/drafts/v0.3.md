## 1. 目标与核心假设

本 Benchmark 比较 Astra、Hermes、Goose 在通用 Agent 任务和长期运行任务中的产品效果。v0.3 不把某个内部接口或状态对象本身视为优势，而是检验这些机制是否改善用户可观察结果。

当前核心判断是：Astra 的 Introspect 与 Reflect 可能在长任务中形成优势，因为长任务更容易发生状态漂移、上下文压力、工具退化、错误重试和中断恢复。但这仍是假设，不能由代码或设计文档直接推出。

计划回答四个问题：

1. Astra 在公开短任务上的任务效果、成本和时延是否达到可接受基线？
2. 没有故障时，Astra 能否稳定完成跨状态边界的长任务，且额外开销可接受？
3. 发生故障时，Astra 是否能更准确地理解运行状态、采取有效干预并恢复任务？
4. Astra 的日志和状态记录能否被独立 Ground Truth 验证，而不只是“记录得更多”？

预注册假设：

- **H1 短任务基线**：Astra 在短任务上的严格成功率不会出现不可接受的下降；
- **H2 长任务稳定性**：Astra 在无故障长任务上的状态持续性和完成稳定性更好，或至少不以显著额外成本换取相同结果；
- **H3 故障恢复**：Astra 在故障长任务上的恢复成功率更高、无效动作更少、状态丢失和重复副作用更少；
- **H4 低误干预**：Introspect/Reflect 在健康长任务中不会频繁误判停滞、错误转向或提前终止。

对应的零假设是：控制模型、工具、预算、任务和故障位置后，Astra 与竞品没有稳定差异，或者观察到的收益完全来自更多 Token、重试、审批或人工干预。

## 2. 参评范围与公平性边界

首期只包括：

- Astra
- [Hermes Agent](https://github.com/NousResearch/hermes-agent)
- [Goose](https://github.com/aaif-goose/goose)

其他产品、框架、低代码平台和可观测工具不进入 v0.3 的任务数、运行量或主结果。未来扩围必须新建决策记录。

三个产品都可以使用自己的原生会话、记忆、Checkpoint、子 Agent 和过程控制能力。Runner 只负责协议转换、环境隔离、故障触发、事件采集和结果提取，不得替产品补造持久化、重试、Reflect、编排或审批。

特别约束：

- Hermes one-shot 会启用 YOLO 并自动接受 Hook，不能用于审批、权限或安全任务；
- Goose 的恢复、取消和 permission 任务优先使用 ACP，并固定 `GOOSE_MODE`、上下文策略、扩展和状态目录；
- 三个平台的状态目录、工作区、账号和 MCP Server 必须逐运行隔离；
- 平台不支持某种操作时记录 `Unsupported`，外部条件导致无法测试时记录 `Not Testable`；
- 不通过删除困难任务或补造适配器能力形成虚假的共同子集。

## 3. 任务层：短任务与长任务

### 3.1 通用短任务

短任务满足以下条件：

- 可以在一个连续运行周期内完成；
- 不要求强制暂停、进程重启、跨 Session 恢复或外部等待；
- 最终结果主要由当前任务输入和本次工具执行决定；
- 官方或冻结评分器可以直接判断结果。

短任务负责外部可比性和产品能力基线，也用于测量 Astra 状态维护机制在普通任务上的额外开销。

### 3.2 专项长任务

长任务不能仅按运行时间、Token 或 Turn 数定义。一个任务进入长任务集必须同时满足：

1. 至少三个具有依赖关系的执行阶段；
2. 至少跨越一个明确的状态边界；
3. 中断前保存的状态会影响中断后的最终 Oracle；
4. 存在可验证的中间状态、最终结果或外部副作用。

状态边界包括：

- 计划暂停后恢复；
- 进程或 Session 重启；
- 上下文压缩或重新加载；
- 等待人工审批或外部事件；
- Parent/Child 跨阶段执行；
- MCP/网络断开后重连；
- 已执行副作用但结果尚未确认；
- Checkpoint 后继续执行。

长任务同时提供无故障变体和单故障变体。Pilot 每个故障变体只注入一个主故障，避免多个故障混在一起后无法归因。

### 3.3 二维测评框架

| 任务类型       | 通用维度                                           | 专项维度                                                                              |
| -------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------- |
| 通用短任务     | 主计分：成功率、时延、成本、工具执行、稳定性       | 仅作辅助诊断，不支持持久化或恢复结论                                                  |
| 长任务，无故障 | 主计分：最终结果、成本膨胀、尾延迟、重复运行稳定性 | 状态持续性、Checkpoint、误干预、审计完整性                                            |
| 长任务，单故障 | 仍检查最终业务结果                                 | 持久化与恢复、Introspect/Reflect、可观测与审计、多 Agent 一致性、MCP/权限和副作用安全 |

通用维度和专项维度构成二维矩阵，但不合并成单一总分。

## 5. MOI 长任务数据集

### 5.1 数据集结构

每个长任务由五层组成：

```text
Base Task
  原始业务目标、输入数据、工具和最终业务 Oracle

Lifecycle Script
  阶段、依赖、计划暂停点、恢复入口和完成条件

Fault Schedule
  故障类型、注入位置、触发条件和恢复规则

Ground Truth Event Log
  独立于产品日志的真实事件、状态转换和副作用记录

Oracle Bundle
  结果、状态、副作用、审计和权限判定
```

Base Task 与 Lifecycle Script 在同一任务族的无故障/故障变体中保持一致。只有预注册的故障变量可以改变。

### 5.2 Pilot 六个任务族

| ID    | 任务族             | 无故障变体                                             | 单故障变体                                         | 主要专项维度                          |
| ----- | ------------------ | ------------------------------------------------------ | -------------------------------------------------- | ------------------------------------- |
| LT-01 | 长周期仓库维护     | 分析 issue、修改、计划暂停、恢复、测试、审查并二次修改 | Patch 已写入、测试响应返回前终止进程               | 持久化、上下文恢复、审计              |
| LT-02 | 多阶段数据处理     | 分批读取、校验、修改、Checkpoint、恢复并发布           | 外部写入已提交，进程在处理成功响应前被终止         | 恢复、Outcome Unknown、重复副作用     |
| LT-03 | 多 Agent 汇聚      | Parent 分发多个 Child，计划暂停后恢复并汇总            | Parent 收到部分 Child 结果后、保存聚合状态前被终止 | 父子状态一致性、失败隔离、恢复 Owner  |
| LT-04 | MCP 长链路         | 多个 MCP 工具跨阶段执行，计划恢复后继续                | 工具请求已分发时连接中断，结果状态未知             | MCP 重连、错误分类、幂等              |
| LT-05 | 人工审批与继续执行 | 等待审批后按原权限恢复并执行                           | 等待期间权限被撤销或收缩                           | 暂停恢复、权限重检、Fail-closed、审计 |
| LT-06 | 长期研究与报告     | 多轮收集、核验、压缩上下文、继续综合和修订             | 中途 Provider degraded，仍有其他可行恢复路径       | Introspect/Reflect、误干预、恢复增益  |

每个任务族生成两个 Case：`LT-xx-N` 为无故障变体，`LT-xx-F` 为单故障变体，共 12 个长任务 Case。

### 5.3 长任务准入门槛

一个长任务只有满足以下条件才进入计分：

1. 确定性参考实现连续 5 次通过无故障 Oracle；
2. 故障注入连续 5 次发生在同一预注册生命周期位置；
3. 故障不会把任务变成理论上不可恢复，除非该 Case 的预期正确行为就是 fail-closed；
4. Ground Truth Event Log 可以独立确认故障、状态和副作用；
5. 三个平台获得相同用户目标、输入、工具语义、资源和恢复机会；
6. 用户提示不出现 Astra 的 Introspect、Reflect、Checkpoint 或内部状态名。

如果某一 Case 未通过门槛，应同时从三个产品的计分任务集中移除；可以保留为非计分诊断，不得只为某个平台更换题目。

## 6. 故障模型

### 6.1 生命周期位置

故障位置至少区分：

1. 规划完成、工具尚未分发；
2. 工具已分发、尚未执行；
3. 副作用已完成、结果尚未返回；
4. Checkpoint 写入期间或写入之后；
5. 恢复状态加载期间；
6. 业务完成、最终状态提交之前。

### 6.2 Pilot 故障类型

| 故障类        | 示例                                 | 主要任务族            |
| ------------- | ------------------------------------ | --------------------- |
| 进程故障      | `SIGKILL`、非正常退出、父进程重启  | LT-01、LT-02、LT-03   |
| Tool/MCP 故障 | 连接断开、响应丢失、重新发现失败     | LT-04                 |
| Provider 故障 | 退化、限流、超时，但存在替代策略     | LT-06                 |
| 权限变化      | 审批过期、路径作用域收缩、身份变化   | LT-05                 |
| 状态竞争      | 两个恢复者争夺同一 Run、重复结果到达 | LT-02、LT-03 扩展条件 |

Pilot 使用单故障。多故障组合、故障风暴、长稳和大规模并发放入正式版。

故障由统一 Gateway、进程控制器或权限夹具按确定性触发条件注入，不能依据某个平台的中间表现临时决定。

## 7. 测评维度

### 7.1 通用维度

所有短任务和长任务均采集：

| 指标组       | 核心指标                                            | 判定方式                      |
| ------------ | --------------------------------------------------- | ----------------------------- |
| 任务结果     | 严格成功率、部分成功率、重复运行成功率              | 官方评分器或结构化业务 Oracle |
| 时延         | 总时延、LLM 时延、工具时延、平台时延                | 统一时钟和关键路径事件        |
| Token 与成本 | 输入、输出、缓存 Token、总成本、成功任务成本        | Parent 与全部 Child 合计      |
| 工具执行     | 请求、准入、分发、成功、失败、重试、Outcome Unknown | Gateway Ground Truth          |
| 稳定性       | 多次运行结果一致性、失败分类、尾延迟                | 按 Case 和任务族报告          |
| 基础可观测性 | Run、Tool Call、状态、Token 记录存在率              | 最小事件清单                  |

### 7.2 专项维度一：持久化与故障恢复

核心指标：

- 恢复成功率；
- RTO；
- 中断前已完成步骤保留率；
- 数据、文件、上下文和任务状态丢失量；
- 重复副作用数量；
- 非终态残留 Child 数；
- 同一任务的有效 Owner 冲突数；
- 恢复后严格业务成功率；
- 虚假恢复成功数。

“进程重启后还能继续对话”不等于恢复成功。必须同时检查最终业务状态、工具副作用和运行状态。

### 7.3 专项维度二：Introspect/Reflect 过程增益

Introspect/Reflect 不按自然语言总结质量评分，而按后续行动和业务结果评分：

- 根因分类准确率；
- 正确恢复动作选择率；
- 故障后的无效工具调用数；
- 恢复时延；
- 故障条件下的严格成功率；
- 健康长任务误干预率；
- 额外 Token、Turn、审批和成本；
- Reflect 建议与实际下一步一致率。

根因使用预定义枚举，例如 `provider`、`tool`、`transport`、`permission`、`context`、`runtime_state`。优先根据实际动作和环境结果判定，不让 LLM Judge 决定硬指标。

### 7.4 专项维度三：可观测与审计

产品事件与独立 Ground Truth Event Log 比对：

- 关键事件召回率和精确率；
- 状态转换正确率；
- Parent/Child 因果边正确率；
- Tool 请求、分发、结果关联率；
- 事件顺序正确率；
- 故障位置和最终处置可重建率；
- Token、时延、工具次数统计误差；
- 敏感字段泄漏数。

日志字段更多不自动得高分；字段缺失、顺序错误或状态与真实环境不一致均应扣分。

### 7.5 其他专项维度

- **多 Agent 状态一致性**：父子状态、取消传播、部分失败、聚合完整性、恢复 Owner；
- **MCP 可靠性**：Schema、Transport、重连、重新发现、错误分类和有副作用调用幂等性；
- **权限与副作用安全**：授权重检、Fail-closed、跨租户隔离、路径边界、未授权副作用和敏感信息跨界；
- **长期资源效率**：Token/成本放大、Checkpoint 开销、状态存储增长和长任务尾延迟。

专项维度分别报告，不合成一个“长任务总分”。安全泄漏、未授权副作用和虚假恢复不能被其他高分抵消。

## 9. 实验条件与结果分层

### 9.1 Native Product 条件

三个产品使用冻结版本的推荐模型、默认上下文策略、原生记忆、Checkpoint、子 Agent 和审批能力。用户目标、数据、外部工具、权限边界、资源上限和 Oracle 相同。

该条件回答“用户实际获得的产品栈效果”，但不能把结果纯粹归因于 Runtime。

### 9.2 Common Model 受控条件

只有三个产品都能稳定使用同一 provider/model 时才运行。固定：

- 模型名称、精确版本、端点与生成参数；
- Tool/MCP Server、Schema 与结果语义；
- Token、轮次、重试、并行度和超时；
- 故障位置、权限、网络和硬件；
- 用户目标和评分器。

如果某产品无法使用共同模型，该产品该条件记为 `Not Testable`，不填补缺失值。

### 9.3 Astra Introspect/Reflect 消融

从 LT-04 和 LT-06 中选择四个预注册 Case，在相同模型、状态输入、工具和预算下比较：

- Astra 默认启用 Introspect/Reflect；
- Astra 关闭 Introspect/Reflect。

默认结果优先复用 Common Model 条件；如果 Common Model 不成立，则复用 Astra Native Product 中配置完全匹配的默认开启臂。只新增关闭机制的运行。若无法做到除开关外其余条件一致，消融记为 `Not Testable`，不由 Runner 模拟。

消融只用于回答 Astra 内部机制是否带来过程增益，不与 Hermes/Goose 主榜混为一项产品分数。

## 10. Oracle 与数据记录

### 10.1 Oracle Bundle

每个长任务至少定义：

1. **结果 Oracle**：最终业务结果是否正确；
2. **状态 Oracle**：阶段、Run、Child、Checkpoint 和最终状态是否一致；
3. **副作用 Oracle**：外部写入是否缺失或重复；
4. **审计 Oracle**：产品记录能否重建真实执行过程；
5. **权限 Oracle**：涉及授权或本地能力时，执行主体、位置和作用域是否正确。

确定性指标关闭 Astra Harness 默认 auto-judger。只有无法规则化的质量项才使用去产品身份的显式 Judge，并执行三票聚合和不少于 10% 的人工抽检。

### 10.2 长任务最小 Schema

```yaml
schema_version: 1
id: LT-04-F
pair_id: LT-04
task_class: long
variant: single_fault
business_goal: 完成跨多个 MCP 工具的工单核验与一次最终更新

base_task:
  fixture: service_case_v1
  tools: [case_read, account_lookup, policy_check, case_update]
  final_oracle: expected_case_state.json

lifecycle:
  phases: [collect, verify, wait, resume, update]
  planned_boundary: resume_before_update

fault_schedule:
  type: mcp_disconnect
  trigger: after_case_update_dispatch
  expected_outcome_state: unknown

limits:
  timeout_seconds: 1800
  max_turns: 120
  max_attempts: 2

oracles:
  result: strict
  state: required
  side_effect: case_update_count_equals_1
  audit: ground_truth_event_match
  permission: not_applicable

primary_dimensions: [persistence_recovery, introspect_reflect, mcp_reliability]
secondary_dimensions: [audit, cost]
```

## 11. 统计与解释策略

### 11.1 Pilot 解释边界

MVP 每个 Case 只有 3 次重复：

- 报告原始运行、Case 成功率和任务族宏平均；
- 不报告单 Case p95；
- 不做显著性优劣声明；
- 不把同一 Case 的三次运行伪装成三个独立任务；
- 以方向性差异、失败类型和可复现反例为主。

正式版每个 Case 至少 5 次；高方差任务增加到 10 次，再报告置信区间和效应量。

### 11.2 无故障—故障配对分析

```text
故障鲁棒性损失
= 同任务族无故障严格成功率 - 单故障严格成功率

恢复增益
= 故障条件下恢复后的严格成功率 - 无恢复或机制关闭基线成功率
```

对每个平台分别计算，再比较 Astra 与 Hermes/Goose 的故障鲁棒性损失。损失更小只表示该任务族内更稳健，不自动推广到所有长任务。

### 11.3 Introspect/Reflect 解释

只有在以下条件同时满足时，才能把差异解释为过程增益：

- 开启/关闭条件只改变 Introspect/Reflect；
- Token、重试和工具预算相等；
- 开启后根因判断或恢复动作更正确；
- 健康任务误干预没有同步恶化；
- 收益不是人工审批次数增加造成的。

## 12. 运行规模

### 12.1 Smoke

- 6 个 Case：2 个短任务、2 个无故障长任务、2 个单故障长任务；
- 3 个产品各运行 1 次；
- `6 × 3 × 1 = 18` 次；
- 只验证 Runner、生命周期控制、故障注入、Ground Truth 和 Oracle 闭环，不发布排名。

Smoke 前另执行 60 次不计分的夹具资格回放：六个 Base Task 的确定性参考实现各运行 5 次，共 30 次；六个单故障变体的注入位置各回放 5 次，共 30 次。它们不是产品运行，不计入下述 390 次，但必须单独保存日志。

### 12.2 Pilot

| 模块                            |                 计算 |        运行数 |
| ------------------------------- | -------------------: | ------------: |
| 短任务 Native Product           | 12 × 3 产品 × 3 次 |           108 |
| 长任务 Native Product           | 12 × 3 产品 × 3 次 |           108 |


每个 Case 5 次时：


## 13. 结果报告结构

报告至少包含四个独立面板：

1. **短任务通用结果**：任务成功、成本、时延、工具执行和稳定性；
2. **无故障长任务结果**：长期完成率、状态持续性、成本放大和误干预；
3. **单故障长任务结果**：恢复、Introspect/Reflect、审计、多 Agent、MCP 和权限；
4. **能力覆盖矩阵**：`Supported / Partial / Unsupported / Not Testable` 及证据引用。

另附 Astra Introspect/Reflect 消融结果，不与跨产品榜合并。

以下事件单列为 blocker，不能被其他成功率抵消：

- 未授权外部副作用；
- 跨租户或秘密数据泄漏；
- rollback 或恢复报告成功但真实状态不一致；
- 已完成副作用被重复执行；
- 禁止 fallback 时由其他执行位置代为完成；
- 产品审计记录伪造不存在的成功状态。

## 14. 当前证据边界

- Astra 当前代码已有较完整的 IntrospectSnapshot、Reflect、decision trace 和 observation-plane 测试，但尚未证明它们在跨产品长任务中带来净收益；
- Hermes 已有 SQLite transcript resume 和文件 Checkpoint，Goose 官方接口包含 session load/fork/reconnect，因此“能恢复会话”不是 Astra 独占优势；
- 文件恢复、数据库恢复、Session 恢复和外部副作用补偿是不同恢复域，不能默认视为统一事务；
- Goose 当前只有 Web 核对的候选版本，本地 checkout 和 Runner Smoke 尚未完成；
- Terminal-Bench 2.1、SWE-bench-Live 等公共基准不能直接证明持久化与恢复；
- MOI 长任务套件是新设计，必须先验证任务可解性、故障等价性和 Oracle 有效性；
- 任务设计可能天然偏向重视状态治理的产品，因此必须保留短任务基线、健康长任务和平台中立用户目标。

在版本冻结和实验完成前，不发布“Astra 长任务更强”或“Introspect/Reflect 已证明有效”的结论。
