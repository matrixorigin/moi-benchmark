# Memoria Agent Memory 评测计划

## 1. 目标与当前范围

第一阶段只实际运行 Memoria，回答两个问题：

1. Memoria 在公开长期记忆数据集上的端到端问答效果如何？
2. Memoria 的 Git-like 记忆管理能力是否正确、稳定且可复现？

当前实验范围：

- **公开数据集**：Memoria 跑 LoCoMo 全量和 LongMemEval-S 全量。
- **特性数据集**：只测试以下三组 Memoria 核心差异化能力：
  1. 快照与回滚；
  2. 分支、Diff 与 Merge；
  3. 矛盾检测、隔离与记忆治理。
- **竞品处理**：暂不重新运行 Mem0 和 Zep；论文或官方公开成绩仅作为外部参考，不作为严格受控的同条件复现结果。

第一阶段不单独测试多 Agent 协同、混合检索和记忆类型系统。多 Agent 协同不是 Memoria 当前明确强调的核心产品能力；混合检索和记忆类型虽然有价值，但不作为本轮差异化重点。

### 2.1 实际运行结果

| 系统 | 数据集 | 结果性质 |
|---|---|---|
| Memoria | LoCoMo 全量 | 本项目实测 |
| Memoria | LongMemEval-S 全量 | 本项目实测或复核已有同事实验 |

### 2.2 外部参考结果

Mem0、Zep 的论文或官网结果可以列入报告，但必须记录：

- 论文或网页地址、发布日期和实验日期；
- 数据集版本与样本范围；
- Reader、Judge、Prompt 和评分方式；
- 检索数量、上下文长度和系统版本；
- 结果属于论文、厂商网页还是第三方复现。

外部结果与 Memoria 实测配置不一致时，只能标注为 `Published reference`，不得表述为严格同条件的性能领先或落后。

Zep 论文使用 LongMemEval-S，并报告：

- Zep + GPT-4o-mini：63.8%；
- Zep + GPT-4o：71.2%。

现有《LongMemEval On Memoria》记录使用 GPT-5.4、Claude Opus 4.6、Claude Sonnet 4.5 作为 Reader，并使用 GPT-5.4 Judge，因此现有 499 题结果不能直接与 Zep 论文分数作严格平台比较。若需要增强可比性，应为 Memoria 增跑一组与 Zep 论文 Reader/Judge 更接近的配置。

## 3. 公开数据集

### 3.1 LoCoMo

- 使用当前目录中的 `datasets/locomo/locomo10.json`；
- 第一阶段运行全量 10 组长对话、5,882 条消息、1,986 道问题；
- 总分之外按 QA category 分组报告；
- 同时明确报告包含 adversarial 类别的全量成绩，以及 Category 1–4 的成绩，便于与只统计 1,540 题的公开结果区分。

### 3.2 LongMemEval-S

- 使用 `datasets/longmemeval/longmemeval_s_cleaned.json`；
- 第一阶段目标为完整 500 题，不再只抽样 100 题；
- 报告以下六类结果：
  - `single-session-user`；
  - `single-session-assistant`；
  - `single-session-preference`；
  - `knowledge-update`；
  - `temporal-reasoning`；
  - `multi-session`；
- Abstention 作为交叉类别单独统计。

现有同事实验共有 500 条 retrieval 记录，其中 `db467c8c` 检索超时，因此 Reader/Judge 实际处理 499 题。正式结果需要优先重试该题并恢复 500 题口径；如果仍然失败，必须将超时计入系统失败，并同时报告 `499 valid samples` 和 `500 total samples` 两种口径。

## 4. Memoria 端到端问答协议

### 4.1 数据写入

Benchmark runner 将原始对话转换为统一消息结构：

```json
{
  "sample_id": "longmemeval-db467c8c",
  "user_id": "longmemeval-db467c8c-user",
  "session_id": "session-01",
  "messages": [
    {
      "role": "user",
      "content": "原始消息文本",
      "created_at": "2023-05-01T09:00:00Z"
    }
  ]
}
```

要求：

- 每个 LoCoMo 对话或 LongMemEval 问题使用隔离的用户/记忆空间；
- 按原始 session 和消息顺序提交；
- 保留原始 speaker、文本和时间信息；
- 不由 Codex 或其他外部 Agent 预先提炼、改写或选择记忆；
- 调用 Memoria 的原始对话观察/写入入口，由 Memoria 完成内部抽取、组织、存储和索引；
- 保存输入 payload hash、请求回执和处理日志。

需要从现有同事实验中补齐并确认：`ingest.py` 实际接口、批次粒度、时间戳映射、抽取模型、Embedding、异步完成判断和失败重试策略。

### 4.2 检索与回答

每道问题执行：

1. 等待 Memoria 的抽取与索引完成；
2. 使用原问题检索相关 memories；
3. 保存原始检索结果和耗时；
4. 固定 Reader 和统一 Prompt 生成答案；
5. 使用数据集官方规则和固定 Judge 评分。

结果分两层记录：

- **Retrieval-only**：支持正确答案的证据是否出现在检索结果中；
- **End-to-end QA**：Reader 使用检索结果后是否回答正确。

现有 LongMemEval 实验采用固定的 Top-10 retrieval snapshot，再让多个 Reader 使用同一份 snapshot。该设计适合隔离 Reader 影响，应继续保留；正式主结果需要事先确定一个固定 Reader，其他 Reader 结果作为消融实验。

## 5. Memoria 特性数据集统一格式

三组特性实验使用确定性的“初始状态 + 操作序列 + 查询 + 预期状态”，而不是主要依赖自然语言问答：

```json
{
  "case_id": "rollback-001",
  "feature": "snapshot_rollback",
  "initial_memories": [],
  "operations": [],
  "assertions": [],
  "expected_state": {}
}
```

每个 case 应保存：

- 原始写入内容及来源；
- 每一步 API 操作；
- 操作前后的 memory ID、版本、状态和 `state_hash`（如果接口提供）；
- 查询结果；
- 明确且可程序化判断的预期结果。

平台能力优先用 API 状态和数据库状态断言，Reader/Judge 只作为补充，避免把平台错误和模型判断错误混在一起。

## 6. 特性实验 A：快照与回滚

### 6.1 基础用例

```text
写入：Alice 住在北京
创建快照 S1
更新：Alice 搬到上海
创建快照 S2
更新：Alice 搬到深圳
验证当前状态为深圳
回滚到 S1，验证状态为北京
回滚到 S2，验证状态为上海
```

### 6.2 覆盖场景

- 单条记忆更新后回滚；
- 新增记忆后回滚；
- 删除记忆后回滚；
- 多条记忆同时变化后回滚；
- 连续创建多个快照并跨级回滚；
- 空状态与空快照；
- 回滚后继续写入；
- 回滚后检索索引是否同步恢复；
- 重复执行同一回滚操作的幂等性；
- 快照不存在、已删除或名称冲突等失败路径。

### 6.3 规模与指标

第一版构造 50 个确定性 case，并生成 100 条带固定随机种子的操作序列。

指标：

- 快照创建成功率；
- 回滚状态正确率；
- memory 数量和内容一致率；
- `state_hash` 一致率；
- 回滚后检索一致率；
- 回滚操作 P50/P95；
- 失败操作后的原状态保持率。

## 7. 特性实验 B：分支、Diff 与 Merge

### 7.1 基础用例

```text
Base：用户喜欢美式咖啡

Branch A：
- 用户最近改喝拿铁
- 用户住在上海

Branch B：
- 用户不再喝咖啡
- 用户住在上海
```

预期：

- Base 不受分支写入污染；
- 两个分支均可独立检索各自状态；
- “住在上海”是相同修改，不应产生冲突；
- “改喝拿铁”和“不再喝咖啡”应进入冲突或明确的合并决策流程；
- Diff 能完整展示新增、更新、删除和冲突；
- Merge 后状态与预期一致，且来源和版本链可追踪。

### 7.2 覆盖场景

- 两个分支修改不同记忆；
- 两个分支写入相同内容；
- 两个分支将同一事实改为不同值；
- 一个分支删除、另一个分支更新；
- 分支嵌套或从非主分支继续分支；
- Merge 后继续写入；
- 冲突 Merge 失败后的状态保持；
- 删除分支后主分支状态不变；
- checkout 前后检索结果与分支状态一致。

### 7.3 规模与指标

第一版构造 30–50 个确定性 case。

指标：

- 分支隔离正确率；
- Diff 完整率和准确率；
- 非冲突 Merge 正确率；
- 冲突检出率；
- Merge 后状态正确率；
- 来源与版本链完整率；
- 失败 Merge 的原子性。

## 8. 特性实验 C：矛盾检测、隔离与记忆治理

该实验直接对应 Memoria 官方强调的 contradiction detection、quarantine、governance 和 audit trail。

### 8.1 数据类型

| 类型 | 示例 | 预期 |
|---|---|---|
| 明确更新 | 住在北京 → 已搬到上海 | 标记旧事实失效或建立演化关系，当前状态为上海 |
| 直接矛盾 | 喜欢咖啡 → 不喜欢咖啡 | 检出矛盾，保留双方来源并进入治理流程 |
| 计划与完成 | 准备搬家 → 已完成搬家 | 区分计划和已发生事实，不误判为普通矛盾 |
| 可并存事实 | 在上海工作；住在苏州 | 不应判为矛盾 |
| 范围变化 | 只周末跑步 → 每天跑步 | 识别条件或范围发生变化 |
| 低置信信息 | “我可能下个月去北京” | 不应等同于已确认事实，可进入低置信状态 |
| 来源冲突 | 用户本人陈述与第三方推断不一致 | 保留来源并按治理规则处理 |
| 多次反转 | 喜欢 → 不喜欢 → 又开始喜欢 | 保留完整演化链，当前状态正确 |

第一版建议构造 100 个 case：

| 类型 | 数量 |
|---|---:|
| 明确事实更新 | 25 |
| 直接矛盾 | 20 |
| 计划/意图/完成状态 | 15 |
| 可并存的伪冲突 | 15 |
| 低置信或不确定表达 | 10 |
| 来源冲突 | 5 |
| 多次反转 | 10 |

### 8.2 操作流程

1. 按时间顺序提交原始对话；
2. 等待抽取、索引和治理任务完成；
3. 获取 active、quarantined、superseded 或其他实际状态；
4. 检查矛盾关系、来源、时间和审计记录；
5. 执行人工确认、纠正、隔离或恢复操作；
6. 再次检查当前状态和检索结果；
7. 对当前事实、历史事实和不可答问题进行少量 QA 验证。

### 8.3 指标

- 矛盾检测 Precision / Recall / F1；
- 伪冲突误报率；
- 低置信记忆隔离准确率；
- stale fact active rate；
- 当前事实状态正确率；
- 历史事实保留率；
- 来源和审计链完整率；
- 治理操作后的状态正确率；
- 治理前后检索结果一致性。

如果当前版本没有稳定、可查询的冲突或隔离状态接口，应先把该部分标为 capability validation，不得只凭最终回答正确就宣称完成了矛盾治理。

### 9.1 公开数据集指标

### 9.3 最终报告

1. Memoria 的 LoCoMo 总体和分类结果；

不把所有维度压缩成单一总分。公开问答质量与平台版本治理能力分别给出结论。

## 10. 公平性与可复现要求

- 固定 Memoria commit、服务版本、部署环境和配置；
- 记录抽取 LLM、Embedding、Reader、Judge 的精确模型版本；
- 固定写入粒度、`top_k`、Prompt、温度和随机种子；
- 保存输入 payload、请求回执、处理状态、检索结果、最终答案和评分明细；
- 每个样本使用隔离空间，失败和重试必须保留记录；
- Reader/Judge 评分应可基于保存结果离线重复执行；
- 特性测试的预期状态在运行前冻结，不根据结果修改；
- 外部公开结果与本地实测结果分栏展示；
- API Key 只通过环境变量读取，禁止写入代码、日志和结果文件。

## 11. 本地资源与费用预算

### 11.1 本地部署判断

第一阶段可以在当前开发机本地完成。当前机器为 Apple M3、8 核 CPU、16 GB 内存、约 94 GB 可用磁盘；官方支持使用 Docker Compose 同时启动 Memoria API 和 MatrixOne。

推荐部署边界：

```text
本机：Memoria API + MatrixOne + Benchmark Runner
外部 API：抽取 LLM + Embedding + Reader + Judge
```

在该部署方式下，本机不加载大模型，主要承担数据库、索引、版本操作和实验调度。资源建议：

| 项目 | 本机建议 | 判断 |
|---|---:|---|
| CPU | 8 核 | 足够功能与质量实验 |
| 内存 | 16 GB；Docker 建议分配约 8 GB | 足够，但 LongMemEval-S 应低并发运行 |
| 磁盘 | 当前可用约 94 GB；实验期间保留至少 30–40 GB 空闲 | 基本足够，需持续监控 MatrixOne 数据和日志 |
| GPU | 不要求 | LLM 与 Embedding 使用外部 API 时不需要 |
| 监控组件 | 默认不启动 Prometheus/Grafana | 节省本机内存 |
| Entity Worker | Smoke 阶段建议从默认 4 降到 2 | 降低内存峰值和 API 并发 |

当前数据文件规模：

| 数据 | 文件大小 |
|---|---:|
| LoCoMo | 约 2.7 MB |
| LongMemEval-S | 约 265 MB |
| LongMemEval Oracle | 约 15 MB |

LongMemEval-S 写入后还会产生原始消息、派生 memories、实体与关系、Embedding、索引、版本和审计数据，因此数据库占用会明显大于 265 MB。正式运行前先执行 10 题和 100 题两级 Smoke，根据 `data/matrixone` 实际增长量外推 500 题磁盘需求。

本地结果可以用于功能正确率和问答质量；检索延迟、吞吐、QPS 等性能数字只能标注为“Mac 本地开发环境结果”。如需生产级性能结论，应使用固定服务器，建议起步配置为 8–16 vCPU、32 GB RAM、200 GB 以上 SSD。只有本地运行 Embedding 或 LLM 时才需要考虑 GPU。

### 11.2 Memoria 本地服务成本

- Memoria 与 MatrixOne 本地运行不产生按请求的云服务费用；
- 使用现有开发机时不新增服务器租赁费用；
- Docker 镜像、数据库文件和日志只消耗本地磁盘；
- 如果迁移到服务器，费用取决于公司现有资源或云厂商报价，本计划暂不写入虚假的固定金额。

### 11.3 LLM 与 Embedding 费用构成

实际费用由四部分组成：

1. Memoria 写入阶段的抽取、实体/事实整理和治理 LLM；
2. Embedding；
3. Reader 回答；
4. Judge 评分。

官方 Compose 当前默认配置包含 `LLM_MODEL=gpt-4o-mini`，Embedding 默认配置为 `BAAI/bge-m3`，但实际费用取决于接入的 API Provider。正式运行必须从 API 响应或Provider账单记录真实 input、cached input 和 output tokens。

以 2026-07-22 可查询到的 OpenAI 标准 API 价格作为预算参考：

| 模型 | 输入价格/1M tokens | 输出价格/1M tokens | 用途示例 |
|---|---:|---:|---|
| GPT-4o mini | $0.15 | $0.60 | Memoria 抽取/整理候选 |
| GPT-5.4 | $2.50 | $15.00 | 现有实验的 Reader/Judge |

GPT-5.4 Batch/Flex 价格通常为标准价格的 50%，但只有在实验脚本实际采用对应处理方式时才能按折扣计算。Embedding 价格不能仅根据模型名推断：`BAAI/bge-m3` 可能是本地模型，也可能由第三方API托管，应按最终Provider单独记录。

价格参考：[GPT-5.4 Model](https://developers.openai.com/api/docs/models/gpt-5.4)、[OpenAI Prompt Caching and GPT-4o mini pricing](https://openai.com/index/api-prompt-caching/)。

费用计算统一使用：

```text
cost = input_tokens / 1,000,000 × input_price
     + cached_input_tokens / 1,000,000 × cached_input_price
     + output_tokens / 1,000,000 × output_price
```

### 11.4 第一轮金额预估

LongMemEval-S 文件约 265 MB，其中包含 500 个问题各自的长历史。按英文文本粗略折算，原始历史约为数千万 tokens；Memoria 内部可能对同一消息执行实体抽取、事实抽取、消歧和矛盾处理，实际 LLM 输入可能是原始文本 token 的数倍。因此在没有 Smoke usage 日志前，不能给出可信的单点价格。

第一轮建议按以下区间预留：

| 费用项 | 初步预算 | 主要不确定性 |
|---|---:|---|
| Memoria 抽取与治理 LLM | $20–$120 | 每条消息触发的 LLM 调用次数、上下文重复、输出长度 |
| Embedding | $0–$20 | 本地还是第三方API、实际向量化文本量 |
| 固定 Reader | $20–$70 | Top-k context长度、所选模型、是否Batch |
| Judge | $10–$40 | Judge Prompt长度、输出和重试 |
| 特性数据集额外 LLM | $5–$20 | 矛盾治理触发次数；快照/分支本身基本不产生LLM费用 |
| 失败重试预留 | 上述费用的20% | 超时、限流和任务重跑 |
| **第一轮总预算** | **建议预留 $80–$300** | 需由Smoke结果收窄 |

该区间覆盖 Memoria 的 LoCoMo 全量、LongMemEval-S 全量和三组特性实验的一次完整运行，不包含本地开发人力成本，也不包含 Mem0/Zep 服务费用，因为本轮不重新运行竞品。

预算收窄方法：

1. 运行 LongMemEval-S 固定 10 题，记录抽取、Embedding、Reader、Judge 的独立 token usage；
2. 再运行固定 100 题，检查线性外推误差和缓存影响；
3. 按实际历史字符数、调用次数和失败率外推 500 题；
4. LoCoMo 单独运行 1 个 conversation 后按10组对话外推；
5. 正式全量运行前设置Provider预算上限和告警。

所有金额均为规划估算，模型价格和Provider报价可能变化；正式运行前必须重新核对官方价格，并在报告中记录价格快照日期。
