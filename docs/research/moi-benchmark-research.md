# MOI Benchmark 竞品、测评维度与数据集调研基线

> 状态：v0.1 研究基线，不是最终测评实施方案  
> 调研日期：2026-07-21  
> 适用范围：MOI 产品中的 Astra 与 NLP2SQL  
> 说明：本文独立建立证据链；`moi-benchmark-plan.md` 只作为问题线索，不作为事实来源。
> 文档角色：研究证据基线；执行约束以 `docs/plans/` 下的计划为准。

## 1. 结论先行

### 1.1 Astra

Astra 应按“可实际使用的 Agent 产品/运行时”测评，而不是只按 Agent 框架测评。当前最合适的直接对标锚点是：

1. **Hermes Agent**：与 Astra 的通用 Agent、终端执行、会话、技能、记忆和多 Agent 方向最接近。
2. **Codex**：代表高成熟度的编码 Agent 产品，以及 CLI、非交互执行、审批、沙箱和应用服务器协议能力。
3. **Goose**：补足开源、模型中立、通用 Agent、CLI/桌面/API、MCP/ACP 的对照。
4. **Cline / OpenCode**：作为模型中立的编码 Agent 产品对照，适合进入扩展轮。

OpenHands、OpenClaw 和 Aider 可作为专项对照；LangGraph、CrewAI、Dify 与 AgentOps 不应进入 Astra 主排行榜，前几者更接近框架/编排或低代码平台，AgentOps 属于可观测性平台。

### 1.2 NLP2SQL

NLP2SQL 必须拆成三个层级，不能把完整产品、算法方法和专用模型混在同一榜单：

- **产品/系统层**：DB-GPT、WrenAI，以及历史对照 Vanna、Chat2DB。
- **方法/Agent 层**：XiYan-SQL、ReFoRCE、CHESS、DAIL-SQL。
- **模型层**：SQLCoder 等专用模型。

其中 Vanna 当前仓库已归档，适合做历史基线；Chat2DB 当前许可证带有附加条件，不能直接当作标准 OSI 开源项目。严格开源主池优先使用 DB-GPT；WrenAI 需要在实施前逐组件复核许可证与可运行边界。

### 1.3 旧方案中需要废弃的前提

以下前提会直接扭曲结论，不应沿用：

- “使用同一个 LLM 后，回答质量、工具选择或 SQL 正确率不再需要比较。”——错误。同模型只能控制一部分变量，检索、上下文构造、工具定义、规划、重试、验证、记忆和安全策略仍会改变最终结果。
- “Astra 的主竞品是 LangGraph、CrewAI、Dify。”——层级错位。它们可以做架构和扩展性参考，但不是 Hermes、Codex 这类完整 Agent 产品的等价对手。
- “NLP2SQL 不需要比较执行准确率。”——错误。端到端 SQL/结果正确性必须是主指标，且优先使用测试套件或结果集判定，不应只看 SQL 字符串完全匹配。
- “所有候选项目可放进一个加权总分。”——暂不成立。产品原生能力、共享模型控制实验和专项能力覆盖应分榜报告，避免用任意权重掩盖差异。

## 2. 研究问题与边界

### 2.1 核心研究问题

在可复现、分层且避免“模型能力等同产品能力”的前提下，哪些开源 Agent 产品与 NLP2SQL 系统最适合作为 Astra 和 MOI NLP2SQL 的对标对象？应由哪些测评维度、公开数据集和自建任务共同支撑可执行 Benchmark？

### 2.2 当前假设

- Astra 被视为面向本地/终端、可审计、可持久化、可调用工具并支持多 Agent 的产品级 Agent 运行时。
- 工作区中尚未发现足以定义 MOI NLP2SQL 完整产品边界的实现文档。因此，本文暂按“自然语言问题 → 上下文/语义检索 → SQL → 执行/校验 → 结果解释”的端到端数据助手定义处理。
- 公开测评用于可比性，自建 MatrixOne/企业任务用于产品相关性；二者不能互相替代。

### 2.3 纳入与排除

纳入：

- 有可访问代码、模型或可复现实验说明；
- 能映射到 Astra 或端到端 NLP2SQL 的实际产品能力；
- 优先采用项目仓库、官方文档、论文和数据集主页等一手来源。

排除出主榜但可作参考：

- 只有宣传材料、无法固定版本或无法获得测试接口的系统；
- 纯基础模型排行榜；
- 与完整 Agent 产品不同层级的编排框架、观测工具；
- 许可证不允许公平复现而又没有可固定的开放版本。

## 3. Astra 竞品盘点

### 3.1 分层候选池

| 层级 | 项目 | 与 Astra 的主要交集 | 主要差异/风险 | 建议 |
|---|---|---|---|---|
| 直接锚点 | [Hermes Agent](https://github.com/NousResearch/hermes-agent) | 通用 Agent、CLI、会话、技能、记忆、工具、多 Agent、MCP、定时任务 | 能力面很宽，默认提示词与运行环境需固定 | MVP 必测 |
| 直接锚点 | [Codex](https://github.com/openai/codex) | 本地编码 Agent、CLI、非交互执行、审批、沙箱、IDE/应用表面 | 主要面向软件工程；模型与产品高度耦合，不能强行纳入同模型赛道 | MVP 必测 |
| 直接锚点 | [Goose](https://github.com/aaif-goose/goose) | 通用 Agent、CLI/桌面/API、模型中立、MCP/ACP | 原生能力与扩展后的能力需分开记录 | MVP 建议必测 |
| 扩展对照 | [Cline](https://github.com/cline/cline) | IDE/CLI、Plan/Act、规则/技能、MCP、自动化 | 编码场景权重更高，产品表面较多 | 第二阶段 |
| 扩展对照 | [OpenCode](https://github.com/anomalyco/opencode) | 模型中立、终端/桌面、计划与子 Agent | 主要是编码 Agent | 第二阶段 |
| 专项对照 | [OpenHands](https://github.com/OpenHands/OpenHands) | 软件工程 Agent、Agent Server、沙箱、自动化 | 产品架构与许可证边界持续变化，应锁版本 | 服务化/软件工程专项 |
| 专项对照 | [OpenClaw](https://github.com/openclaw/openclaw) | 通用个人 Agent、消息渠道、多 Agent 路由 | 与 Astra 的终端主路径不完全相同 | 消息、多租户和安全专项 |
| 窄基线 | [Aider](https://github.com/Aider-AI/aider) | 终端编码、Git 工作流 | 单一结对编程路径，能力范围窄 | 编码任务下限基线 |
| 非主榜 | LangGraph / CrewAI | 图式或角色式 Agent 编排 | SDK/框架，不是等价产品 | 仅做架构参考 |
| 非主榜 | Dify | 低代码 LLM 应用/工作流平台 | 部署、交互和抽象层不同 | 仅做平台参考 |
| 非主榜 | AgentOps | 轨迹与可观测性 | 不负责完成同类用户任务 | 仅做可观测性参考 |

### 3.2 为什么 MVP 选择 Hermes + Codex + Goose

这三个项目形成互补三角：

- Hermes 提供与 Astra 最接近的“通用 Agent 产品宽度”；
- Codex 提供成熟的编码执行、审批、沙箱和自动化参照；其 [Rust CLI 文档](https://github.com/openai/codex/blob/main/codex-rs/README.md) 明确包含非交互 `exec` 与沙箱运行方式，[App Server 文档](https://github.com/openai/codex/blob/main/codex-rs/app-server/README.md) 则提供协议和审批事件参照；
- Goose 提供模型中立的开放实现，可降低只比较“某一家模型+产品整体”的偏差。

这不是认定三者功能完全等价，而是用最少数量覆盖三种关键设计取向。扩展轮再加入 Cline/OpenCode，验证结论是否对其他模型中立编码 Agent 仍成立。

### 3.3 Astra 测评赛道

不能把所有结果压成一个数字，至少应分三个赛道：

| 赛道 | 目的 | 模型策略 | 可比较的结论 |
|---|---|---|---|
| A. 原生最佳实践 | 衡量用户实际能获得的产品价值 | 使用各项目官方推荐的模型、提示和运行方式；锁版本 | “产品整体在其最佳实践下能完成什么” |
| B. 同模型控制 | 尽量隔离运行时、编排和工具层的影响 | 仅纳入能接入同一模型/provider 的产品；统一温度、预算和任务接口 | “相近模型条件下，产品层造成什么差异” |
| C. 能力与失效专项 | 测持久化、安全、恢复、多 Agent、审计等非纯结果能力 | 按能力选择固定模型；注入可控故障 | “产品是否具备并可靠实现关键工程能力” |

Codex 若不能与其他项目使用同一模型，不应被伪装成 B 赛道的等价样本；它仍可进入 A 和 C 赛道。三个赛道分别报告，不合并总分。

## 4. Astra 测评维度

### 4.1 一级维度与可观测指标

| 一级维度 | 核心问题 | 建议指标 |
|---|---|---|
| 任务成功 | 是否完成了用户真正要求的结果 | 任务通过率、测试通过率、输出工件正确率、部分完成率、未知结果率 |
| 轨迹与工具 | 是否通过可靠、可解释的路径完成 | 工具调用成功率、无效调用率、重复步骤、错误后策略变化、不可逆副作用、轨迹完整度 |
| 效率 | 为成功付出了多少资源 | 端到端时延 p50/p95、首次有效动作时延、模型 token/成本、工具调用数、CPU/内存、成功任务成本 |
| 鲁棒与恢复 | 环境出错后能否恢复而非静默失败 | 模型/工具/网络/进程故障恢复率、重试收益、断点续跑率、重复副作用率、状态一致性 |
| 安全与权限 | 是否把影响范围限制在用户授权内 | 越权操作率、审批命中/绕过率、提示注入攻击成功率、秘密泄漏率、目录/租户隔离、沙箱逃逸 |
| 会话、记忆与长程状态 | 跨轮次、跨进程是否保留正确状态 | 事实保留率、错误记忆率、长程任务完成率、会话恢复率、记忆污染和隔离 |
| 多 Agent | 委派是否真正带来收益且可控制 | 委派成功率、并行加速、聚合正确率、取消传播、子任务失败隔离、重复工作率 |
| 可审计与可复现 | 是否能说明“发生了什么、为什么” | 事件覆盖率、因果链完整度、用量归因、决策/审批记录、重放一致性、版本与配置可追溯 |
| 扩展与协议 | 能否接入现有工具和控制面 | MCP/ACP/API/无头模式覆盖、技能安装成功率、协议一致性、扩展故障隔离 |
| 部署与运维 | 是否能稳定进入真实环境 | 安装成功率、冷启动时延、资源占用、升级/回滚、配置复杂度、自托管能力、许可证约束 |

### 4.2 指标原则

- 主成功指标优先使用单元测试、环境状态、文件差异、结构化规则和真实工具返回，不以 LLM Judge 为默认裁判。
- 如果开放任务必须使用 Judge，应报告 Judge 模型、提示词、重复次数、一致性和人工复核抽样。
- 每次失败必须区分：模型推理失败、工具失败、产品编排失败、环境失败和评分器失败。
- “未能判定”必须单列，不能自动记为成功或失败。
- 成本以“成功任务成本”与“所有尝试成本”同时报告，避免失败系统因早停而看起来便宜。

## 5. Astra 数据集与任务组合

单一公开基准无法覆盖 Astra 的产品边界。建议采用“公开任务结果 + MOI 工程能力任务”组合。

### 5.1 公开层

| 数据集/框架 | 覆盖能力 | 采用方式 | 局限 |
|---|---|---|---|
| [Terminal-Bench 2.0](https://github.com/harbor-framework/terminal-bench-2) | 真实终端任务、容器化执行、可验证结果 | 作为通用终端执行核心集，先跑可在统一基础设施复现的子集 | 不能覆盖会话持久化、多租户和产品 UI |
| [SWE-bench](https://github.com/swe-bench) / [SWE-bench-Live](https://swe-bench-live.github.io/) | 真实软件仓库问题修复 | 以 Live 或未污染任务优先；Verified 子集用于兼容对照 | 污染、环境构建和任务成本高；偏编码 |
| [GAIA](https://huggingface.co/gaia-benchmark) | 通用助理的搜索、推理和工具组合 | 选可在相同工具边界下执行且答案可核验的子集 | 对产品工程能力覆盖不足 |
| [tau2-bench](https://github.com/sierra-research/tau2-bench) | 有政策、工具与用户交互的任务 | 评估澄清、合规和多轮工具使用 | 客服领域与 Astra 主场景存在差距 |
| [MCP-Universe](https://mcp-universe.github.io/usage.html) | 多 MCP 服务、执行式评估 | 用于 MCP 工具发现、参数选择和跨服务任务 | 依赖服务版本和外部环境 |
| [AgentBench](https://github.com/THUDM/AgentBench) | OS、数据库、知识图谱、Web 等多环境 | 作为覆盖型基线，选择仍能稳定运行的环境 | 较老任务可能被训练数据污染或失去区分度 |
| [AgentDojo](https://agentdojo.spylab.ai/) | 工具型 Agent 的提示注入与安全 | 安全专项必测 | 不衡量完整的正常任务体验 |

### 5.2 MOI 自建层

公开集普遍缺少以下能力，必须自建：

1. **持久化与恢复**：长任务中杀进程、重启服务、切换会话，检查状态、工具副作用和审批是否一致。
2. **可审计性**：从用户请求追踪到规划、模型调用、工具调用、权限决策、成本和最终工件，检查事件是否完整关联。
3. **多 Agent 生命周期**：委派、并行、汇总、取消、超时、子 Agent 崩溃与重复提交。
4. **权限和租户隔离**：跨目录、跨用户、跨项目读取/写入，秘密注入与工具返回中的恶意指令。
5. **定时与后台任务**：触发时间、幂等、错过触发后的补偿、并发冲突和失败告警。
6. **MatrixOne 状态一致性**：会话、事件、检查点、记忆和用量记录的一致性与可恢复性。

建议每类先设计 10–20 个可重复场景，使用故障注入矩阵生成变体，而不是一次性手写大量不可维护任务。

## 6. NLP2SQL 开源项目盘点

### 6.1 分层候选池

| 层级 | 项目 | 可比较内容 | 状态/边界 | 建议 |
|---|---|---|---|---|
| 产品/系统 | [DB-GPT](https://github.com/eosphoros-ai/DB-GPT) | 数据源接入、NL2SQL、Agent/工作流、图表与报告、沙箱、多模型 | 能力面宽，需要关闭无关能力并固定配置 | 严格开源主池 |
| 产品/语义层 | [WrenAI](https://github.com/Canner/WrenAI) | 语义上下文、业务定义、查询与治理、多数据源 | 当前项目形态已演进；组件许可证和可运行表面需逐项复核 | 产品/语义层专项 |
| 方法/模型生态 | [XiYan-SQL](https://github.com/XGenerationLab/XiYan-SQL) | Schema 表达、中文、方言、候选生成/精排/修复 | 不是与完整产品完全等价的 UI/治理系统 | 算法主基线 |
| 方法/Agent | [ReFoRCE](https://github.com/Snowflake-Labs/ReFoRCE) | 大 Schema 压缩、Schema Linking、自修正、候选一致性 | 面向 Spider 2.0 的方法基线 | 企业复杂任务基线 |
| 方法/Agent | [CHESS](https://arxiv.org/abs/2405.16755) | 检索、Schema 选择、候选生成、单元测试 | 论文方法；复现需固定代码、模型与数据 | 方法参考/可复现后入榜 |
| 历史产品 | [Vanna](https://github.com/vanna-ai/vanna) | RAG 式 NL2SQL、权限/审计与 Web 交互设计 | 仓库已归档；只能锁定历史版本 | 历史基线，不代表当前活跃生态 |
| 有条件产品 | [Chat2DB](https://github.com/OtterMind/Chat2DB) | 数据库客户端、自然语言查询、可视化 | 当前 [许可证](https://github.com/OtterMind/Chat2DB/blob/main/LICENSE) 含附加条件；社区版和商业版边界需核查 | 排除严格开源主榜或锁定合规版本 |
| 专用模型 | [SQLCoder](https://github.com/defog-ai/sqlcoder) | SQL 生成模型能力 | 不是端到端产品；权重许可证需单独审查 | 模型下限/上限参考 |
| 方法 | [DAIL-SQL](https://arxiv.org/abs/2308.15363) | 示例选择、提示构造、候选选择 | 方法而非产品 | 消融参考 |

### 6.2 建议对照结构

不要建立一个混合榜单。至少建立：

1. **产品系统榜**：MOI NLP2SQL、DB-GPT、可复现的 WrenAI 表面；历史 Vanna 单列。
2. **算法流水线榜**：MOI 核心流水线、XiYan-SQL、ReFoRCE，以及复现条件满足后的 CHESS。
3. **模型控制实验**：在相同模型、Schema 文本、最大 token 和数据库权限下，比较不同检索、规划、校验与修复策略。
4. **原生最佳实践实验**：允许各系统使用推荐模型和语义配置，衡量用户最终价值。

产品榜不能因一个方法在 Spider/BIRD 上成绩好就判定其产品能力更强；方法榜也不能因产品带有权限或可视化功能就获得 SQL 正确性加分。

## 7. NLP2SQL 测评维度

| 一级维度 | 核心指标 | 诊断指标 |
|---|---|---|
| 端到端正确性 | 测试套件执行准确率、结果集正确率、任务成功率 | SQL 可执行率、部分结果正确率、非确定结果率 |
| Schema Linking | 表召回/精确率、列映射、Join Key 正确率 | 漏表、冗余表、错误关联、错误粒度 |
| 业务语义 Grounding | 指标定义、时间口径、维度、过滤条件正确率 | 业务术语检索、同义词、单位和时区错误 |
| 值 Grounding | 实体值、枚举、日期、范围解析正确率 | 模糊匹配、拼写变化、中文时间表达、NULL 语义 |
| 歧义与不可回答 | 澄清成功率、正确拒答率、错误自信率 | 歧义类型识别、澄清轮数、置信度校准 |
| 多轮交互 | 上下文一致率、跟进修改成功率、纠错成功率 | 指代、约束累积、撤销条件、话题切换 |
| 鲁棒性 | Schema/NL/SQL 扰动后性能下降、跨方言迁移 | 大 Schema、脏数据、命名变化、缺失元数据 |
| 校验与修复 | 首次成功率、修复后成功率、净增益 | 错误分类、无效重试、语义错误发现、重试预算 |
| 安全与治理 | DDL/DML 阻断、越权查询率、敏感数据泄漏率 | 注入、行列权限、租户隔离、全表扫描与资源滥用 |
| 效率 | 时延 p50/p95、token/成本、数据库执行成本 | Schema 检索耗时、重试数、扫描量、超时率 |
| 可审计性 | 问题→上下文→SQL→执行→答案链路完整率 | 版本、权限、证据、改写和错误的可追溯性 |

### 7.1 正确性判定优先级

1. **业务任务断言/测试用例**：能验证结果和业务约束时优先。
2. **结果集等价**：在隔离、只读数据库上执行候选 SQL 与金标准/断言。
3. **Test Suite Accuracy**：使用多数据库实例减少“错误 SQL 碰巧同结果”。官方 [test-suite-sql-eval](https://github.com/taoyds/test-suite-sql-eval) 提供比字符串完全匹配更可靠的基础。
4. **结构化 SQL 比较**：用于定位错误，而不是唯一主指标。
5. **LLM Judge**：只用于解释质量或无法完全规则化的业务语义，并需要人工抽检。

## 8. NLP2SQL 数据集组合

### 8.1 数据集角色

| 数据集 | 主要价值 | 建议位置 | 关键局限 |
|---|---|---|---|
| [Spider 1.0](https://github.com/taoyds/spider) | 经典跨库兼容性、工具链成熟 | 烟雾测试和历史对照 | 老旧、污染风险高、企业复杂度不足，不做 headline |
| [BIRD](https://bird-bench.github.io/) | 更真实的数据库内容、外部知识、效率 | 开发主集；优先 Mini-Dev/可审计子集 | 标注质量需抽检，完整集运行成本较高 |
| [Spider 2.0](https://spider2-sql.github.io/) | 企业级大 Schema、多方言、多步工作流 | 企业复杂任务核心层 | 环境重、云数仓成本和凭据管理复杂 |
| [BEAVER](https://beaverbench.github.io/) | 来自企业查询日志，含表检索、Join、列映射等诊断标注 | Schema Linking 与企业诊断核心层 | 私有测试限制本地错误分析；领域分布需与 MOI 对齐 |
| [LiveSQLBench](https://livesqlbench.ai/) | 动态、降低污染、面向真实场景 | 外部盲测/发布门禁 | 动态版本必须记录快照，长期复现难度更高 |
| [BIRD-INTERACT](https://bird-interact.github.io/) | 歧义、用户交互、BI/CRUD、多轮与可执行测试 | 交互与澄清核心层 | 用户模拟器会引入额外方差 |
| [SParC](https://yale-lily.github.io/sparc) | 跨轮上下文 SQL | 低成本多轮回归 | 较老、交互复杂度有限 |
| [CoSQL](https://yale-lily.github.io/cosql) | 对话、澄清和不可回答 | 多轮诊断 | 较老，数据库与表达分布有限 |
| [Dr.Spider](https://github.com/awslabs/diagnostic-robustness-text-to-sql) | 17 类扰动的鲁棒性诊断 | 鲁棒专项必测 | 基于 Spider 生态，仍需企业 Schema 扰动 |
| [CSpider](https://taolusi.github.io/CSpider-explorer/) | 中文单轮兼容性 | 中文烟雾层 | 主要是 Spider 的中文化，不代表中文企业语义 |
| [CHASE](https://aclanthology.org/2021.acl-long.180/) | 中文多轮、上下文依赖 | 中文交互核心层 | 仍需补制造/企业术语和 MatrixOne 方言 |
| [PRACTIQ](https://github.com/amazon-science/conversational-ambiguous-unanswerable-text2sql) | 歧义、不可回答和澄清 | 交互安全专项 | 与产品用户模拟方式需对齐 |

### 8.2 推荐分层执行

| 层 | 目的 | 数据集组合 | 发布频率 |
|---|---|---|---|
| L0 烟雾 | 检查接口、方言和评分器 | Spider 1.0 小集 + CSpider 小集 + 自建基本 SQL | 每次提交 |
| L1 快速回归 | 发现常见正确性和 Schema 问题 | BIRD Mini-Dev/修订子集 + CHASE 子集 | 每日/主干合并 |
| L2 企业复杂 | 验证大 Schema、业务流程、方言 | BEAVER 公共集 + Spider 2.0 Lite/固定子集 | 每周/版本候选 |
| L3 交互与鲁棒 | 歧义、澄清、多轮、扰动 | BIRD-INTERACT Lite + PRACTIQ + Dr.Spider | 每周/版本候选 |
| L4 MOI 私有盲测 | 产品相关性、污染控制、MatrixOne 与业务语义 | 制造业 Schema、业务指标、权限攻击、真实问法 | 发布门禁；严格隐藏 |

### 8.3 标注与污染控制

公开 Text-to-SQL 数据集存在标注错误与陈旧 SQL 的系统性风险。一个近期的 [跨数据集审计项目](https://github.com/uiuc-kang-lab/text_to_sql_benchmarks) 已专门提供修订版本与错误分析，因此：

- 在正式采用 BIRD/Spider 派生集前，对开发子集做人工和执行双重抽检；
- 记录数据集 commit、数据库快照、评分器 commit 和修订补丁；
- 不用开发集反复人工调到“全会”，保留隐藏测试与动态集；
- 对外报告公开集与私有集结果，不公布私有 SQL、Schema 映射和完整自然语言问法；
- 同一问题的翻译版本不能被当作独立证据累加。

## 9. 实验设计约束

### 9.1 共同控制项

每个实验至少记录：

- 项目 commit/tag、依赖锁文件和容器镜像；
- 模型/provider/版本、采样参数、上下文上限和推理预算；
- 系统提示、工具说明、技能/插件、MCP Server 版本；
- 操作系统、CPU/GPU/内存、网络策略、数据库快照；
- 最大轮数、最大工具调用、超时、重试和人工介入规则；
- 每个任务的原始事件、标准输出、工具输入输出、最终工件和评分器结果。

### 9.2 重复与统计

- 非确定性任务至少重复 3 次；高方差任务建议 5 次以上。
- 报告均值/中位数、置信区间和任务级配对差异，不只报告单次总分。
- 对失败类别做分层 bootstrap 或任务级配对检验；不要把性质完全不同的数据集样本简单相加。
- 版本升级时同时跑固定回归集与新鲜盲测集，分别测回归和泛化。

### 9.3 公平性

- “同模型”不等于“同能力边界”：工具数量、上下文容量、提示模板和原生优化必须单独披露。
- 原生能力被关闭后的结果只能解释为控制实验，不能替代真实用户价值。
- 允许扩展的系统必须区分“开箱即用”和“安装额外插件/自写适配器”结果。
- 人工介入若允许，必须记录次数、时点和提供的信息；无人值守结果单列。

## 10. 第一阶段最小可行方案

### 10.1 Astra MVP

- 参评：Astra、Hermes、Codex、Goose。
- 任务：Terminal-Bench 2.0 固定子集、SWE-bench-Live/Verified 固定子集、AgentDojo 安全集、自建恢复/审计/多 Agent 各 10 个场景。
- 赛道：原生最佳实践；共享模型控制（仅兼容产品）；工程专项。
- 输出：任务成功率、成功成本、时延、故障恢复、安全、状态一致性和审计覆盖七张分表，不产生单一总冠军。

### 10.2 NLP2SQL MVP

- 产品参评：MOI NLP2SQL、DB-GPT；WrenAI 完成边界和许可证复核后加入。
- 方法参评：MOI 核心流水线、XiYan-SQL、ReFoRCE。
- 数据：BIRD Mini-Dev/修订子集、BEAVER 公共子集、Spider 2.0 Lite/固定子集、BIRD-INTERACT Lite、Dr.Spider、CHASE 子集、MOI 私有 MatrixOne 集。
- 输出：端到端正确性为主，Schema Linking、歧义/澄清、鲁棒、安全、效率和审计为诊断分表。

## 11. 证据冲突与处理

| 表面冲突 | 解释 | 处理方式 |
|---|---|---|
| Codex 很强，但不能共享同一模型 | 原生产品价值与同模型因果控制是两个研究问题 | 进入原生/专项赛道，不强塞同模型榜 |
| Hermes、Goose 是通用 Agent，Codex/OpenCode 偏编码 | 竞品边界确实跨场景，完全同质化会丢失产品定位 | 共享终端/工具任务做核心集，场景专属任务单列 |
| DB-GPT/WrenAI 是产品，ReFoRCE/CHESS 是方法 | 产品能力与 SQL 算法能力不是同一测量对象 | 产品榜、方法榜和模型控制实验分开 |
| 公开集可比较，但容易污染；私有集贴近业务，但不可复核 | 外部可比性与内部有效性存在张力 | 同时报告公开固定集、动态集和私有盲测，不混为一个分数 |
| SQL 字符串不同，但结果可能等价 | Exact Match 会误判合法等价 SQL | 优先测试套件/结果集/业务断言，结构匹配只做诊断 |
| 最新项目能力强，但接口和许可证变化快 | 快速演进会破坏长期可复现性 | 锁 commit、镜像和许可证快照；每季复核候选池 |

## 12. 仍需确认的产品问题

这些问题不会阻止研究基线，但会改变最终权重和任务分布：

1. Astra 的首要用户场景是通用知识工作、软件工程、数据运维，还是三者的组合？
2. Astra 的多 Agent、定时任务、消息渠道、技能市场与 MatrixOne 持久化中，哪些属于近期正式交付能力？
3. MOI NLP2SQL 当前入口、支持的数据源/方言、语义层、权限模型和输出形态是什么？
4. 是否允许把外部系统接入相同模型/provider，还是只能测各自原生配置？
5. 私有企业 Schema、真实问法和业务指标是否可在隔离环境中用于隐藏测试？
6. 最终用途是产品选型、研发回归、对外市场结论，还是三者都需要？三类用途应有不同的报告口径。

## 14. 证据说明与限制

- 本文优先引用项目官方仓库、官方数据集主页和原始论文；项目状态与许可证属于高变化信息，正式执行前必须再次核验。

- 没有完整安装并运行全部候选项目；“可测”表示公开资料显示存在相应接口，不等于已在 MOI 基础设施中验证成功。
- 公开排行榜结果因模型、提示、工具、预算和环境不同，不能直接复制为 MOI 结论。
- 本文没有声称穷尽所有开源项目；候选池优先覆盖与目标产品最相关且可复现的设计取向。
