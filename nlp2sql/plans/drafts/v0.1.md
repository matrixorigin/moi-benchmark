# MOI NLP2SQL 产品能力评测计划

> 状态：Draft v0.1，待确认数据库范围、参评系统和排期后冻结
>
> 评测对象：MOI 与可连接 MatrixOne/MySQL 协议数据源的 NLP2SQL 产品
>
> 评测数据：MatrixOne 现有真实业务表及约 10 条已有问题
>
> 评测规模：建立 50～60 条人工验真的私有测试集，完成 MOI 与 2～3 个竞品的可复现产品横评

## 1. 评测目标

本轮评测回答以下问题：

1. 面对同一套真实业务表和用户问题，MOI 与竞品能否生成语义正确、可执行的 SQL，并返回正确结果？
2. 各产品在 Schema 理解、业务术语、复杂查询、自动纠错、澄清与拒答方面有什么差异？
3. 各产品达到正确结果所需的时间、模型调用、SQL 执行次数、Token 和人工配置成本是多少？
4. 产品是否能够在只读、最小权限条件下安全运行，并留下可追溯的 SQL 和错误记录？

本评测只报告当前测试范围内的真实产品能力，不把公开榜单成绩、模型论文成绩或不同数据库上的结果混入主排名。

## 2. 范围与非目标

### 2.1 评测范围

- 使用 MatrixOne 中已存在的若干真实业务表；
- 将同事已有约 10 条问题整理为开发/回归种子；
- 构建约 50～60 条经过执行验证和人工复核的 MVP 测试集；
- 使用 MOI、Wren AI Cloud、Chat2DB 客户端和 XiYan GBI 完成约 10 条 Smoke 测试；
- 根据 Smoke 结果选择 2～3 个竞品进入 50～60 条正式评测；
- Vanna、DB-GPT 仅在确认有增量价值时本地运行，不提前申请专用服务器；
- 以最终执行结果正确性为主指标，同时记录产品能力、性能、成本和安全性。

### 2.2 非目标

- 不全量迁移 Spider、BIRD 等公开数据集；
- 不比较仅有论文分数、无法在同一数据上实测的方法；
- 不把 SQL 字符串完全一致作为主指标；
- 不把数据库执行性能等同于 NLP2SQL 生成能力；
- 不在未经数据安全确认的情况下把真实业务数据接入外部 SaaS；
- 不根据某个产品的测试结果反向修改隐藏测试集或 Golden。

本计划不引入公开数据集；如将来单独开展公开基准研究，应另行立项并冻结数据集和方言迁移规则。

## 3. 评测对象与接入方式

### 3.1 候选参评系统

| 系统 | 定位 | 本计划中的角色 | 数据接入方式 |
|---|---|---|---|
| MOI | 被测产品 | 必测 | 原生连接 MatrixOne |
| Wren AI Cloud | 托管式语义层/GenBI 产品 | 参加 Smoke，连通且合规时进入正式评测 | 网页端连接公网可达的 MatrixOne/MySQL 测试库 |
| Chat2DB | 带 AI 能力的数据库客户端 | 参加 Smoke，优先进入正式评测 | 使用桌面客户端从公司网络直连 MatrixOne |
| XiYan GBI | 阿里云托管数据分析产品 | 参加 Smoke，连通且合规时进入正式评测 | 网页端连接经审批、云端可达的 MySQL 协议测试库 |
| Vanna OSS | NLP2SQL 产品/框架 | 前三者无法形成 2～3 个有效竞品时作为替补 | 在个人电脑启动服务，通过浏览器访问本地 UI |
| DB-GPT | 数据库问答与 Agent 平台 | 前三者无法形成 2～3 个有效竞品时作为替补 | 在个人电脑或测试机启动 Web 服务 |
| XiYan-SQL | 中文 Text-to-SQL 方法/模型 | 不纳入本产品评测 | 如需测试，另建方法评测 |

参评名单、产品版本、许可证、部署方式、模型和连接方式均需在正式运行前冻结。

### 3.2 数据连接优先级

优先采用所有产品连接同一个 MatrixOne 只读测试快照。网页云产品只有在网络和数据安全均允许时才直接连接；公司内网数据库优先使用桌面客户端或个人电脑本地服务。

```text
MOI ───────┐
Chat2DB ───┼──→ 内网 MatrixOne 测试快照
本地产品 ──┘

Wren Cloud ─┐
XiYan GBI ──┴──→ 经审批的公网/VPC 可达脱敏快照
```

直连前逐项验证：

- MySQL 驱动握手和查询是否正常；
- `information_schema`、`SHOW TABLES`、`SHOW CREATE TABLE` 是否能被产品正确读取；
- 表名、列名、类型、注释、主外键和视图是否识别完整；
- 产品是否依赖 MatrixOne 未提供的 MySQL 系统表或元数据；
- 生成 SQL 是否使用 MatrixOne 支持的方言和函数。

只有竞品无法稳定直连 MatrixOne 时，才把固定数据快照迁移到 MySQL 或 PostgreSQL。迁移后必须校验表结构、行数和 Golden 查询结果等价；迁移问题归类为环境或适配问题，不计为产品生成错误。

### 3.3 零服务器优先策略

本计划不预先申请专用服务器和 GPU：

1. MOI 使用现有产品环境；
2. Wren AI 使用 Cloud 网页版；
3. Chat2DB 使用桌面客户端；
4. XiYan 使用 GBI 网页试用；
5. Vanna、DB-GPT 暂不部署，仅在前三个竞品覆盖不足或无法接入时作为替补。

这一策略覆盖本计划的产品体验和人工评测。只有后续需要持续自动回归时，才评估 API 套餐或一台 8 核、16～32 GB 内存的共享测试机；使用外部模型 API 时不需要 GPU。

费用与约束：

| 项目 | 本计划预算口径 |
|---|---|
| Wren AI Cloud | 先使用免费额度；API 自动化需要支持 API 的付费计划及独立 credits |
| Chat2DB | 使用 30 天/200 次 AI 试用；超出后购买订阅或自行配置模型 API |
| XiYan GBI | 使用 200 问免费试用；正式套餐约 299/499 元每月，具体以运行时官方价格为准 |
| Vanna/DB-GPT OSS | 软件免费；如启用则承担模型 API 和本地运行成本 |
| 统一模型 API | Smoke 后按实际 Token 外推；预留 500～1,000 元机动额度，不代表必须消耗 |

网页方案的限制是模型、重试和调用轨迹不一定完全可控。报告必须披露这些差异，不把不同内置模型下的结果解释为纯算法对比。

## 4. 统一评测口径

本计划只设一套产品评测口径：各产品使用官方推荐的网页、客户端、语义层、Schema 检索和自动修复能力，同时尽可能统一其外部条件。

共同约束：

- 使用同一数据库快照、表范围和等价只读权限；
- 提供同一来源、语义等价的表说明、业务术语和示例；
- 能配置模型时使用同一模型、温度和最大输出长度；不能配置时记录产品内置模型为不可控变量；
- 统一单题最大等待时间和人工重试规则；
- 不允许为某个产品单独加入答案特定的提示或示例；
- 正式运行前冻结配置，隐藏测试期间不逐题调参。

每个产品必须记录：

- 人工配置时间；
- 表/字段说明数量；
- 业务规则和示例 SQL 数量；
- 使用的模型及调用预算；
- 产品是否自动读取数据样例、保存历史问题或持续学习；
- 模型、重试或调用轨迹中无法观测的项目。

## 5. 数据库准备

### 5.1 数据快照与隔离

1. 选定本轮涉及的数据库、表和视图；
2. 固定数据快照时间，评测期间不允许业务写入改变结果；
3. 为每个产品创建权限等价的只读账号；
4. 设置查询超时、最大返回行数和并发限制；
5. 记录表结构、行数、数据时间范围和必要的数据哈希；
6. 将连接串和凭证保存在安全配置中，不写入 Git 或运行报告。

建议优先使用脱敏的测试副本。若必须使用真实环境，需确认不会把 Schema、样例值、SQL 结果或日志上传至未经批准的第三方服务。

### 5.2 Schema 与业务知识清单

为测试范围内的表建立统一的数据字典：

| 内容 | 示例 |
|---|---|
| 表说明 | `orders` 表示订单主表 |
| 字段说明 | `paid_amount` 表示实际支付金额 |
| 表关系 | `orders.customer_id = customers.id` |
| 枚举值 | `status = 3` 表示已完成 |
| 时间口径 | 订单月份按 `paid_at` 而非 `created_at` 计算 |
| 业务指标 | 销售额仅统计已支付且未退款订单 |
| 权限规则 | 分析账号不可读取手机号等敏感字段 |

各产品获得语义等价的知识；允许转换成产品要求的 YAML、DDL、文档或配置格式，但不能改变信息量。

## 6. 私有测试集构建

### 6.1 现有 10 条问题的处理

逐条补齐：

- 原始问题和真实业务意图；
- 人工确认的 Golden SQL；
- SQL 执行结果或结果哈希；
- 涉及的表、字段、JOIN 和业务口径；
- 难度、能力标签和问题来源；
- 是否有歧义、是否可回答；
- 数据快照和审核人。

如果这些问题已经用于 MOI 开发、Prompt、few-shot 或竞品配置，则归入 `dev/regression`，不能进入隐藏测试集。

### 6.2 候选问题生成方式

采用两种方式混合构建：

1. **SQL-first，约 60%～70%**：先设计真实、可执行且有区分度的 SQL，再写成自然语言问题；
2. **Question-first，约 30%～40%**：从业务人员、报表、历史 SQL、工单或真实分析需求收集问题，再由数据人员编写 Golden SQL。

LLM 可以根据脱敏 Schema 和能力矩阵生成候选问题、候选 SQL 和标签，但候选不能直接进入正式测试集。

### 6.3 MVP 能力矩阵

测试集目标 50～60 条，建议分布如下：

| 类型 | 建议题量 | 主要能力 |
|---|---:|---|
| 单表过滤、排序、Top-K | 6～8 | 字段和值理解 |
| 聚合、分组、占比 | 8～10 | 聚合与粒度 |
| 两表/多表 JOIN | 10～12 | Schema Linking 和 JOIN 路径 |
| 时间查询 | 6～8 | 时间字段、范围、同比/环比 |
| 复杂 SQL | 6～8 | 子查询、CTE、窗口、排名 |
| 业务指标与口径 | 5～6 | 业务术语、状态和计算规则 |
| 歧义与澄清 | 3～4 | 有效澄清而非猜测 |
| 不可回答/越权请求 | 3～4 | 正确拒答和安全性 |

同时覆盖简称、同义词、中英混合、NULL、空结果、重复值、枚举状态、相似字段名和多日期字段。

### 6.4 题族与防泄漏

同一语义的改写、参数替换和难度变体使用同一个 `semantic_id`。同一题族不能跨越开发集和隐藏测试集。

建议切分：

- `dev`：现有约 10 条种子及少量扩展，用于适配和配置；
- `test`：30～40 条隐藏单轮问题；
- `challenge`：10 条左右复杂、歧义、不可回答或安全问题；
- `regression`：由已确认的线上失败持续沉淀，不追溯修改历史结果。

## 7. Golden 验证与样本准入

每条正式样本必须通过：

1. Golden SQL 能在冻结的 MatrixOne 快照上执行；
2. 问题中的条件、字段、时间和业务口径均被 SQL 覆盖；
3. SQL 没有加入问题未要求的条件；
4. JOIN 不产生意外重复计数；
5. 结果不是由错误过滤导致的偶然空集；
6. 没有 `ORDER BY` 时，结果不依赖数据库返回顺序；
7. 时间、浮点数、NULL、大小写和枚举口径已明确；
8. 至少一名数据/业务人员完成审核，关键业务指标建议双人复核。

若同一问题存在多种合理解释，应改写消除歧义，或明确标记为“应澄清”，不能强行保留唯一 Golden。

## 8. 测试用例格式

```yaml
case_id: nl2sql_0001
semantic_id: monthly_sales_by_region
split: test
question: 上个月各地区的销售额是多少？
language: zh-CN
database_snapshot: mo_nl2sql_20260722
answerable: true
expected_action: execute_read
difficulty: medium
skills: [join, group_by, date_filter, business_metric]
tables: [orders, customers]
business_context_ref: context_v1
gold:
  matrixone_sql: SELECT ...
  alternative_sql: []
  result_hash: sha256:...
  expected_columns: [region, sales_amount]
comparison:
  row_order: ignore
  float_tolerance: 0.000001
review:
  status: approved
  reviewer: reviewer_id
```

私有问题、完整 Schema、Golden SQL 和业务规则不得提交到公开仓库。Git 中只保留格式、生成规则、Manifest 和脱敏示例；真实 payload 存放在受控位置。

## 9. 执行协议

每个系统按统一流程执行：

1. 重置会话，加载冻结配置；
2. 提交一个测试问题；
3. 保存产品返回的 SQL、澄清问题、自然语言答案和错误；
4. 使用只读账号执行最终 SQL；
5. 将结果与 Golden 结果比较；
6. 记录模型调用、SQL 执行、重试、时延、Token 和成本；
7. 标记成功、产品失败或环境失败；
8. 清理会话，避免前一题泄漏到下一题。

默认预算建议：

- 每题最多 3 次模型调用；
- 每题最多 2 次 SQL 修复执行；
- 单条 SQL 超时 60 秒；
- 最大返回 1,000 行；
- 温度为 0 或产品允许的最低值；
- 正式测试期间禁止人工修改 SQL。

具体预算在 Smoke 阶段验证后冻结。

## 10. 评分方法

### 10.1 主指标

| 指标 | 定义 |
|---|---|
| Execution Accuracy | 最终 SQL 结果与 Golden 结果等价的样本比例 |
| First-pass Success | 第一次生成、未经修复即结果正确的比例 |
| Final Success | 在固定重试预算内最终正确的比例 |
| Clarification Success | 应澄清问题中提出有效澄清并完成任务的比例 |
| Correct Abstention | 不可回答或越权问题中正确拒答的比例 |
| Safe Task Success | 结果正确且无权限、写入或资源违规的比例 |

### 10.2 诊断指标

- SQL 语法有效率和可执行率；
- 表、字段和 JOIN 路径正确率；
- 时间、过滤、聚合粒度和业务口径正确率；
- 自动修复触发率与净增益；
- 错误自信执行率；
- P50/P95 端到端时延；
- 平均模型调用数、SQL 执行数、Token 和单题成本；
- 初始配置人时和人工操作步骤数；
- SQL、错误和运行链路的可审计完整率。

### 10.3 结果等价规则

- 主评判依据是执行结果，不是 SQL 字符串；
- 没有 `ORDER BY` 时按无序集合比较；
- 明确列顺序、重复行、NULL、浮点容差和时间格式；
- 对大结果集可比较规范化后的内容哈希，同时抽样核验；
- 空结果必须结合约束和变异数据检查，避免错误 SQL 偶然通过；
- 无法评分的环境、适配器或 Golden 问题单列为 `Not Scorable`，不能静默删除。

## 11. 失败分类

```text
DATA_OR_GOLDEN
DATABASE_OR_ENVIRONMENT
CONNECTOR_OR_METADATA
PRODUCT_RUNTIME
SCHEMA_LINKING
VALUE_GROUNDING
BUSINESS_GROUNDING
SQL_SYNTAX
SQL_SEMANTIC
VALIDATION_OR_REPAIR
CLARIFICATION_OR_ABSTENTION
SAFETY_OR_PERMISSION
UNKNOWN
```

只有证据足够时才归因；否则标记为 `UNKNOWN`。数据库连接或迁移失败不能记为 NLP2SQL 生成失败。

## 12. 公平性与安全约束

1. 所有产品使用同一数据快照和等价权限；
2. 开发集和隐藏测试集按题族隔离；
3. 正式评测前冻结配置，之后不按单题修补提示；
4. 记录产品自动保存的历史问题，防止跨系统或跨轮污染；
5. 正常题只允许 `SELECT` 或只读等价操作；
6. DDL/DML、提示注入和越权测试只在隔离测试库中进行；
7. 出现未授权写入、跨租户泄漏或敏感数据外发时立即停止该配置；
8. 外部 SaaS 接入真实数据前必须完成数据安全确认和脱敏；
9. 报告明确区分实测结果、外部公开结果和未验证能力。

## 13. 实施阶段

### Phase 0：准备、连通性与 Smoke，预计 3.25 人天

- 盘点 MatrixOne 表、关系、字段说明和数据权限；
- 冻结测试快照和只读账号；
- 申请 Wren AI Cloud、Chat2DB 和 XiYan GBI 的试用账号；
- 确认测试数据是否允许连接云端产品；不允许时准备脱敏副本或取消对应云产品；
- 验证 MOI、Wren AI Cloud、Chat2DB 和 XiYan GBI 对测试库的连接和元数据识别；
- 整理现有约 10 条问题及 Golden；
- 用现有约 10 条问题完成网页/客户端 Smoke 和结果比较器验证；
- 根据可连接性、有效结果和人工操作成本，选择 2～3 个竞品进入正式评测。

退出条件：所有必测系统均能完成至少一个可评分问题，或明确记录连接阻塞及备选迁移方案。

### Phase 1：测试集构建与冻结，预计 2.75 人天

- 根据能力矩阵生成候选问题；
- 完成 SQL-first 与 Question-first 问题补充；
- 执行、去重、分级并人工审核；
- 冻结 50～60 条测试集和题族划分。

退出条件：所有正式样本均通过 Golden 准入检查，开发集和隐藏集无题族泄漏。

### Phase 2：产品横评与报告，预计 4.25 人天

- 使用开发集完成有限配置和联调；
- 冻结各系统版本、模型、上下文、预算和连接配置；
- 按统一评测口径运行全部参评产品；
- 对 10～15 条关键或不稳定样本运行 3 次，其余样本运行 1 次；
- 复核所有环境错误、安全事件、系统分歧和 `UNKNOWN` 失败；
- 输出主结果表、成本表和典型失败案例。

### 持续维护

- 将稳定高价值样本沉淀为回归集；
- 根据线上失败持续补充新题，但不改写历史结果；
- 数据或业务口径变化时发布新快照和测试集版本。

### 人天拆分

以下按 1 名主执行人员、50～60 条测试、MOI + 2～3 个正式竞品估算：

| 工作项 | 主执行人天 | 配合人天 | 说明 |
|---|---:|---:|---|
| 数据范围、权限和脱敏确认 | 0.5 | 0.5 | 需要数据库/安全负责人确认 |
| 表结构、关系和业务字典盘点 | 0.75 | 0.5 | 依赖熟悉业务数据的同事 |
| 现有约 10 题 Golden 整理与执行验证 | 0.5 | 0.25 | 已被用于调试的题归入 dev |
| 三个竞品账号、网络和元数据连通 | 1.0 | 0.25 | 不含审批等待时间 |
| 10 题 Smoke、筛选正式竞品 | 0.5 | 0.0 | 网页/客户端人工执行 |
| 能力矩阵和候选问题生成 | 0.75 | 0.25 | SQL-first 与真实问题混合 |
| 40～50 条新增 Golden SQL 编写与验证 | 1.5 | 0.75 | 每条执行验证，关键业务题双人复核 |
| 去重、难度标注、题族切分和冻结 | 0.5 | 0.25 | 防止 dev/test 泄漏 |
| 竞品配置冻结和开发集联调 | 0.5 | 0.0 | 禁止按隐藏测试逐题调参 |
| 50～60 题正式人工运行与留痕 | 2.0 | 0.0 | 按 MOI + 2～3 个竞品估算 |
| 关键/不稳定样本复跑 | 0.5 | 0.0 | 10～15 题每个产品运行 3 次 |
| 评分、失败归因和人工复核 | 0.75 | 0.25 | 复核全部环境错误和结果分歧 |
| 结果报告和评审修改 | 0.5 | 0.25 | 包括成本、限制和典型案例 |
| **合计** | **10.25** | **3.0** | **总投入约 13.25 人天** |

主执行工作约 10.25 人天，数据库、安全和业务同事合计约 3 人天。账号审批、网络开放和采购可能增加 2～5 个自然日，但不计入上述人天；单人主执行建议按 2～3 个自然周安排。

本计划优先人工网页/客户端运行。若后续需要持续回归，再单独评估 API Adapter 自动化，不把这部分预先计入当前人天。
