# MOI 文档解析 Benchmark 方案

## 一、目标与范围

评估 **MOI 解析后端 vs MinerU / Paddle / 其他竞品** 在公开数据集和私有行业 case 上的解析能力。

整体分为两条评测线：

```text
公开数据集 benchmark
私有数据集 benchmark
```

公开数据集用于获得可复现、可与公开结果对比的官方口径分数；私有数据集用于验证复杂行业文档和真实客户场景中的解析效果。

---

## 二、公开数据集 Benchmark

### 2.1 目标

得到 MOI 在公开 Benchmark 上的官方口径分数，并与竞品公开分数或同环境重跑分数进行对比。

### 2.2 主要工作

#### 1. 选择公开数据集

优先采用 OmniDocBench，后续可补充以下专项 Benchmark：

- olmOCR-Bench
- DocLayNet
- PubTables-1M

#### 2. 下载和整理数据集

整理并版本化保存：

- PDF/图片等原始输入
- 官方 Ground Truth
- 官方 Evaluator
- 配置文件
- 运行说明
- 数据集版本和校验信息

#### 3. 运行 MOI 解析后端

使用 MOI 后端解析公开数据集的全部文档，并保存：

- 原始输出
- 运行日志
- 失败样本
- 处理耗时
- CPU、内存、GPU、显存等资源占用

#### 4. 编写公开集 Adapter

将 MOI 的 `parse.json` 映射成 OmniDocBench 官方评分器要求的 Prediction 格式：

```text
MOI parse.json -> OmniDocBench prediction
```

Adapter 只负责格式转换和必要的无损归一化，不得修正解析结果本身。

#### 5. 运行官方评分器

使用 OmniDocBench 官方 Evaluator 计算：

```text
Overall Edit
Text Edit
Formula Edit
Table Edit
Reading Order Edit
Table TEDS
Formula CDM
```

#### 6. 收集竞品公开分数

收集 MinerU、Paddle、olmOCR、Marker 等产品在 OmniDocBench 上的公开分数，并记录：

- 结果来源
- 产品/模型版本
- 运行配置
- 数据集版本
- 评测时间
- 是否为官方结果

#### 7. 对比和报告

初版可以采用 MOI 自测分数与竞品公开分数对比。是否在同一环境中重新运行 MinerU、Paddle 等竞品，取决于项目时间和资源安排；如果时间允许，应通过同环境重跑减少数据版本、硬件环境和运行配置差异对结论的影响。如果时间不允许，则使用竞品公开分数，并在报告中明确标注来源、版本、配置差异和不可直接等同比较的限制。

### 2.3 输出目录

```text
public-benchmark/
  omnidocbench/
    moi_raw_outputs/
    moi_predictions/
    official_scores/
    competitor_public_scores.md
    report.md
```

## 三、私有数据集 Benchmark

### 3.1 目标

评估 MOI 和竞品在行业特殊 Case 及真实业务文档上的解析表现。

### 3.2 主要工作

#### 1. 整理私有 Case

选择一批有代表性的行业文档，重点覆盖：

```text
复杂表格
跨页表
公式
图文 caption
多栏阅读顺序
页眉页脚
标题层级
扫描件/OCR
特殊行业术语
```

#### 2. 建立私有数据集目录

每个 Case 保留原文件、Parser 输出、Annotation、Golden 和版本信息：

```text
datasets/private/cases/<case-id>/
  source.pdf
  golden.json
  annotation.json
  parses/
    moi/
    mineru/
    paddle/
```

#### 3. 运行所有 Parser

对同一批私有文档分别运行：

```text
MOI
MinerU
Paddle / PP-StructureV3
olmOCR / Marker / 其他
```

每次运行保存原始输出、日志、失败样本、处理耗时和资源占用。

#### 4. 编写私有集 Adapter

将不同 Parser 的原始输出统一映射到内部 Parse Blocks Schema：

```text
MOI output     -> MOI parse blocks
MinerU output  -> MOI parse blocks
Paddle output  -> MOI parse blocks
```

#### 5. 编写和校正 Golden

可以使用某个 Parser 的输出生成 Draft Golden，但必须经过人工校正和审核后，才能成为正式 Golden。不能直接将 MOI 或任一竞品的输出作为真值。

Golden 需要覆盖：

```text
header_footer
title_hierarchy
layout
table
table_teds
image
caption
text_content
formula
reading_order
```

#### 6. 使用内部评分器跑分

使用 `parsing_benchmark` 对所有 Parser 输出进行统一评分：

```text
parser output blocks vs private golden.json
```

#### 7. 输出私有集报告

报告至少包括：

```text
总体 F1
各维度 P/R/F1
Case 级分数
错误明细
失败样本
解析耗时
资源成本
```

### 3.3 输出目录

```text
private-benchmark/
  cases/
  golden/
  parser_outputs/
  normalized_outputs/
  scores/
  report.md
```

## 四、统一 Runner 和报告

为了使公开集和私有集两条评测线可复现、可维护，需要建设统一调度层。

### 4.1 Parser Runner 抽象

每个 Parser 提供独立 Runner：

```text
run_moi
run_mineru
run_paddle
run_olmocr
```

Runner 负责启动解析任务、保存原始输出、捕获异常并采集性能数据，不负责结果修正或评分。

### 4.2 Adapter 抽象

提供两类 Adapter：

```text
public adapter:
  parser output -> OmniDocBench prediction

private adapter:
  parser output -> MOI parse blocks
```

### 4.3 Score Runner

提供两类 Scorer：

```text
official scorer:
  OmniDocBench evaluator

internal scorer:
  parsing_benchmark
```

### 4.4 统一 Metadata

每次运行和评分必须记录：

```text
parser 名称
parser 版本
parser 配置
dataset 版本
case 数量
成功/失败数量
硬件环境
平均每页耗时
峰值内存/显存
git commit
command
run_id
```

建议同时记录原始输出和 Prediction/Normalized Output 的文件校验值，以支持结果追溯。

### 4.5 最终报告结构

公开分和私有分不硬合成一个总分，分别报告：

```text
1. Executive Summary
2. Public Benchmark: OmniDocBench
3. Private Industry Benchmark
4. Per-dimension Analysis
5. Performance / Cost
6. Failure Cases
7. Conclusions
```

## 五、实施优先级

1. **先完成公开集 OmniDocBench 路径**
   快速获得 MOI 的公开 Benchmark 分数。

2. **同步收集竞品公开分数**
   Paddle 文档已有 OmniDocBench 分数；继续确认 MinerU、olmOCR、Marker 等是否在官方文档、论文或 README 中提供公开结果。

3. **建立 20～50 个私有行业 Case**
   覆盖真实客户痛点，第一阶段重代表性而不是数量。

4. **完成私有 Golden 校正**
   人工审核后的 Golden 是私有 Benchmark 可信度的核心。

5. **编写 MinerU/Paddle Adapter**
   使竞品输出能够进入内部统一评分器。

6. **视时间安排进行竞品同环境重跑**
   竞品同环境重跑属于增强项，不作为第一阶段的强制交付。时间允许时，在初版报告验证价值后补充 MinerU、Paddle 等竞品的同环境结果；时间不足时采用有明确来源和版本信息的公开分数，并披露比较限制。

### 1. 公开 Benchmark 分数


### 2. 竞品公开分数整理

```text
MinerU / Paddle / olmOCR / Marker 等公开结果和来源
```

```text

```

## 七、API 执行资源与费用估算

### 7.1 执行方式

第一阶段优先通过官方 API 运行 MinerU 和 Paddle 文档解析，不要求本地部署竞品模型。API Token/Access Token 仅用于身份认证，不等同于大模型 Token；实际额度和费用通常按解析页数计算。

统一执行流程：

```text
OmniDocBench images（1,651 页）
  -> MinerU 官方 API
  -> Paddle / 百度云文档解析 API
  -> 保存原始响应、任务 ID、日志和耗时
  -> Adapter 统一格式
  -> OmniDocBench 官方评分器
```

### 7.2 所需资源

采用云 API 时，本地不需要为 MinerU/Paddle 准备推理 GPU，主要需要：

- 一台能够稳定访问 API 的执行机。
- 约 4～10 GB 本地磁盘空间，用于数据集、API 原始响应、图片/Markdown/JSON 结果、日志和中间文件。
- MinerU API Token。
- PaddleOCR 官方 API Access Token，或百度智能云 OCR Access Token。
- 支持异步任务提交、轮询、限速、失败重试和断点续跑的 Runner。
- API 调用记录，包括 `task_id`、提交时间、完成时间、状态码、重试次数和计费页数。

MOI 后端仍按现有部署方式运行，其计算资源和内部成本单独记录，不与竞品 API 费用混算。

### 7.3 OmniDocBench 公开集费用估算

估算基于当前本地数据集的 **1,651 个页面样本**，价格和免费额度以 **2026-07-22** 查询到的官方信息为参考，正式执行前必须再次确认控制台实际报价和账号额度。

| 项目 | 当前额度/价格参考 | 1,651 页预算估算 | 说明 |
|---|---|---:|---|
| MinerU 精准解析 API | Token 免费申请；账号提供每日最高优先级页数额度，超额任务可能降优先级 | **预计 ¥0** | 可拆分到不同日期执行；如果官方后续启用收费或账号额度不同，以控制台为准 |
| MinerU Agent 轻量 API | 无需 Token，但文件/页数限制更严格且受 IP 限频 | **预计 ¥0** | 固定轻量模型，不适合作为 MinerU 高质量能力的主要对比结果 |
| 百度云文档解析 / PaddleOCR-VL | 个人或企业认证账号提供 200 页免费测试额度 | 剩余计费页数约 **1,451 页** | 免费额度需确认账号是否已领取或消耗 |
| 百度云标准资源包 | 当前 1,000 页约 ¥90 | **约 ¥180** | 需要购买两个 1,000 页资源包；约有 549 页剩余额度 |
| 百度云新人活动价 | 当前活动最低约 ¥9/千页 | **约 ¥18** | 活动资格、购买上限和有效期需在下单前确认 |

预算建议：

```text
最低预算：约 ¥18
保守预算：约 ¥180～¥200
MinerU：暂按 ¥0 预算
```

上述预算仅包含竞品解析 API，不包含：

- MOI 内部解析后端的算力成本。
- Adapter 和 Runner 的开发人力。
- API 失败重试可能产生的重复计费。
- 私有数据集及后续补充数据集的调用费用。
- 数据存储、网络和人工标注成本。

### 7.4 费用与调用风险控制

- 正式全量执行前，先用 10～20 页进行 API 冒烟测试，确认输出格式、模型版本和实际扣费规则。
- Runner 必须使用确定性 `sample_id` 和任务状态记录，避免重复提交造成重复计费。
- 为每个 API 配置最大页数、最大重试次数和预算上限。
- MinerU/Paddle 必须记录实际调用的模型版本，不能仅记录产品名称。
- 保留控制台额度截图或账单导出，最终报告使用实际计费金额替换预算值。
- 若 PaddleOCR 官方托管 API 与百度云文档解析属于不同服务入口，应分别记录服务名称和计费口径，不能混用价格。

## 八、风险与应对

### 8.1 新旧版本 MOI 解析质量不确定

目前无法确认新版 MOI 解析后端相较老版的解析质量是提升、持平还是退化。如果直接使用新版结果参与公开和私有 Benchmark，可能出现以下风险：

- 新版在部分解析维度上发生质量回退，导致 Benchmark 结果低于老版。
- 新旧版本的输出 Schema、默认参数或处理链路发生变化，造成结果不可直接比较。
- 在未建立版本基线前，难以判断问题来自模型能力、工程改动、Adapter，还是评分链路。

建议在正式对外比较前增加一轮 MOI 新旧版本回归验证：

```text
同一批代表性样本
  -> MOI 老版
  -> MOI 新版
  -> 使用同一 Adapter 和 Scorer 比较
```

至少覆盖文字、表格、公式、阅读顺序、标题层级和复杂行业 Case，并记录新旧版本、配置、输出 Schema 和逐 Case 差异。如果时间有限，可以先选取 20～50 个代表性 Case 做快速回归；在质量差异确认前，报告中应明确标注该风险，避免将新版结果直接解释为 MOI 稳定能力上限。

### 8.2 竞品公开分数的可比性风险

当时间不足、无法同环境重跑竞品时，公开分数可能受到产品版本、数据集版本、运行配置和硬件环境差异影响。此类结果可用于趋势参考，但不应表述为严格的同条件横向结论。

应在报告中为每项竞品结果记录来源及限制，并区分：

```text
Official published score
Third-party published score
Internally reproduced score
```

## 九、关键原则

- 公开 Benchmark 优先复用官方数据、官方配置和官方 Evaluator。
- Adapter 只做格式映射和无损归一化，不修正 Parser 的错误。
- 私有 Golden 必须经过人工校正和审核，并记录版本及变更历史。
- 所有 Parser 使用同一数据输入、相同资源等级和明确记录的配置。
- 公开分与私有分分别呈现，不合并成含义不清的单一总分。
- 所有运行结果必须能够通过版本、配置、命令、日志和 `run_id` 复现。
